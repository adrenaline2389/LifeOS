# LifeOS v1.2 Agent Prompt

## 目标

根据当前项目文档，实现 LifeOS v1.2。

v1.2 是第一个真实子系统闭环版本。目标是在 v1.1 的六系统启动面板基础上，开启“个人生态系统”的最小观察闭环。

实现过程应支持多 agent 协作。主 Agent 负责整体集成、任务分派、进度跟踪和最终验证；子 Agent 按 `doc/v1.2/tasks/` 中的任务文件领取模块。

## 输入

必须先阅读：

1. `doc/current.md`
2. `doc/README.md`
3. `doc/decisions/`
4. `doc/v1.2/design.md`
5. `doc/v1.2/tasks/progress.md`
6. `doc/v1.2/tasks/`

必要时参考：

- `doc/v1.1/design.md`
- `doc/v1.1/tasks/`
- `doc/roadmap/v1.2.md`

参考 v1.1 时，只继承仍然有效的本地优先、技术栈、启动面板、六系统主体、onboarding 定位、模块边界和测试原则。不要把 v1.1 中“所有子系统都只是入口卡片”的范围限制带入 v1.2 的个人生态系统。

## 输出

完成 LifeOS v1.2 的可运行实现。

v1.2 实现后应具备：

- v1.1 的首次启动扫描流程。
- v1.1 的 LifeOS 启动面板。
- 可进入的个人生态系统页面。
- 个人生态系统六个维度。
- 选择性记录单个维度的生态观察点。
- 同一次进入个人生态系统页面时，同一维度重复选择会覆盖本次进入刚生成的观察点。
- 用户可以删除误触生成的生态观察点。
- 本地保存、读取和重置生态观察点。
- 当前生态状态面板。
- 今日生态轨迹面板。
- 生活晴雨表面板。
- 1 天、7 天、15 天、30 天范围切换。
- 从个人生态系统返回启动面板。

v1.2 实现后不应具备用户侧：

- 健康建议。
- 医疗建议。
- 心理诊断。
- AI 分析。
- 显性数字评分 UI。
- 连续打卡、连续天数、完成率或惩罚。
- JSON 导出入口。
- Markdown 导出入口。
- 完整个人说明书编辑入口。
- 其他五个子系统的真实业务闭环。

## 技术要求

- 应用框架：Next.js。
- 前端框架：React。
- 开发语言：TypeScript。
- 本地数据：IndexedDB。
- IndexedDB 封装：Dexie。
- 测试：沿用当前项目测试方案。
- 包管理器：沿用当前项目配置。

必须通过：

```bash
npm run test
npm run typecheck
npm run lint
```

最终发布前还必须通过：

```bash
npm run build
```

不要引入：

- 后端服务。
- 账号系统。
- 云同步。
- 云端数据库。
- 外部 AI 服务。
- 用于分析、推理或诊断用户的 AI 能力。

## 核心产品约束

LifeOS v1.2 必须保持本地优先：

- 用户数据只保存在用户自己的设备上。
- 默认不向任何云端上传个人生活数据。
- 不实现账号系统、云同步或多人协作。
- 不使用 AI 分析、AI 推理、心理诊断或人格诊断。
- 初始扫描、生态观察和生活晴雨表都由确定性规则生成。

个人生态系统必须保持非审判式：

- 用户选择语义状态，系统内部可以换算分数。
- UI 不展示内部分数。
- UI 不把平均状态描述为诊断或建议。
- UI 不要求用户每天完成记录。
- UI 不使用连续打卡、完成率或惩罚文案。
- 用户进入个人生态系统时可以只查看，不记录。

界面语言使用简体中文。

## 模块边界

严格按照 `doc/v1.2/design.md` 和 `doc/v1.2/tasks/` 中的模块职责实现。

### App Shell

负责高层流程切换：

- 无本地扫描结果时展示 onboarding。
- 有本地扫描结果时展示 LifeOS 启动面板。
- 可以从启动面板进入个人生态系统。
- 可以从个人生态系统返回启动面板。
- 重置后回到首次启动状态。

不包含生态维度配置、生态聚合规则或 IndexedDB 细节。

### Startup Dashboard

负责展示 LifeOS 启动面板：

- 继续展示六个子系统。
- 个人生态系统入口可进入。
- 个人生态系统被推荐时，推荐卡片可进入个人生态系统。
- 其他五个子系统继续后续开放。
- 不展示 JSON / Markdown 导出入口。
- 不展示完整个人说明书编辑入口。

### Personal Ecosystem

负责个人生态系统页面：

- 展示页面标题 `个人生态系统`。
- 展示系统定义 `生理基础与生活环境`。
- 展示当前生态状态。
- 展示今日生态轨迹。
- 展示生活晴雨表。
- 允许用户选择性记录某一个维度。
- 记录控件使用一条七档语义轴表达程度差异。
- 同一页面会话里，同一维度重复记录时覆盖本次会话内该维度的上一条记录。
- 今日生态轨迹中的观察点可以删除。

### Ecosystem Dimensions

负责生态维度配置：

- 六个维度固定顺序为睡眠恢复、作息节律、身体状态、饮食饮水、活动舒展、环境支撑。
- 每个维度恰好 7 个语义值。
- 每个语义值映射 -3 到 +3 的内部状态。
- `没观察` 不进入语义值列表。

### Ecosystem Insights

负责确定性聚合：

- 当前生态状态。
- 今日生态轨迹。
- 生活晴雨表。
- 本地时间范围。
- 平均状态映射回语义摘要。

不包含 React UI、IndexedDB 读写、AI 分析或诊断文案。

### Local Data

负责本地保存、读取和重置：

- 保存 onboarding 回答。
- 保存启动扫描结果。
- 保存个人生态系统观察点。
- 删除个人生态系统观察点。
- 按时间范围读取观察点。
- 清空本地 LifeOS 数据。

不作为用户侧能力提供 JSON 或 Markdown 导出。

## Agent 工作方式

主 Agent 负责：

- 阅读 v1.2 全部输入文档。
- 检查当前代码与 v1.2 目标的差异。
- 按 `doc/v1.2/tasks/` 分派子 Agent。
- 确保并行任务不会互相覆盖。
- 审核子 Agent 输出。
- 集成模块。
- 更新 `doc/v1.2/tasks/progress.md`。
- 运行最终验证。

子 Agent 负责：

- 只处理领取的任务文件。
- 补充或更新对应测试。
- 保持模块边界清晰。
- 不修改无关模块。
- 不引入云端依赖。
- 不重复定义核心类型。
- 不把个人说明书、导入导出、AI 能力或诊断能力重新引入 v1.2。

## 并行开发建议

推荐顺序：

1. 先完成生态维度配置与类型。
2. 同步或随后完成本地生态观察点存储。
3. 再完成生态聚合与生活晴雨表规则。
4. 再实现个人生态系统页面 UI。
5. 再接入 App Shell 与启动面板入口。
6. 最后做文档同步和全量验收。

## 完成标准

v1.2 完成时必须满足：

- `doc/v1.2/tasks/progress.md` 所有任务完成。
- 所有测试通过。
- TypeScript 类型检查通过。
- lint 通过。
- build 通过。
- 本地页面可完成首次扫描。
- 完成扫描后进入 LifeOS 启动面板。
- 点击个人生态系统可以进入个人生态系统页面。
- 个人生态系统展示六个维度。
- 用户可以只记录一个维度。
- 记录后当前生态状态更新。
- 今日生态轨迹展示当天观察点。
- 生活晴雨表支持 1 天、7 天、15 天、30 天。
- UI 不展示内部分数。
- UI 不强制用户记录。
- UI 没有连续打卡、完成率或惩罚文案。
- 其他五个子系统仍显示后续开放。
- 用户界面没有 JSON 导出。
- 用户界面没有 Markdown 导出。
- 用户界面没有完整个人说明书编辑入口。
- 重置后可以回到首次启动状态，并清空生态观察点。
