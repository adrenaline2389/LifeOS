# LifeOS v1.2 设计文档

## 1. 版本定位

LifeOS v1.2 是第一个真实子系统闭环版本。

v1.1 已经完成减法重构：它把 LifeOS 从“个人说明书原型”纠偏为“六个子系统驱动的本地人生操作系统启动器”。v1.2 不继续扩展 onboarding，也不恢复个人说明书、导出或云端能力，而是在 v1.1 的启动面板基础上真正开启第一个子系统。

v1.2 的目标是：

> 建立个人生态系统的最小观察闭环。

这个闭环不是健康管理、习惯打卡或诊断系统。它帮助用户在本地记录和回顾身体与生活环境是否支撑自己运行。

## 2. 继承自 v1.1 的原则

v1.2 继续继承这些原则：

- 本地优先。
- 用户数据保存在用户自己的设备上。
- 不使用账号系统。
- 不使用云同步。
- 不使用云端数据库保存个人生活数据。
- 不使用 AI 分析、AI 推理、心理诊断、人格诊断或医疗建议。
- 技术栈继续使用 Next.js + React + TypeScript。
- 本地持久化继续使用 IndexedDB + Dexie。
- 界面语言使用简体中文。
- 保留复古 Macintosh 风格和旧电脑感。
- onboarding 仍然只是初始扫描和子系统推荐器。
- 个人说明书仍然是后期板块，不是当前主功能。
- 推荐、扫描线索和生态回顾都必须是确定性逻辑，不得伪装成诊断结论。

v1.2 不继承 v1.1 中“六个子系统都只是入口卡片”的范围限制。v1.2 会把个人生态系统推进为可使用的真实子系统；其他五个子系统仍保持后续开放。

## 3. v1.2 新增内容

### 3.1 个人生态系统最小闭环

v1.2 新增个人生态系统页面。

最小闭环是：

```text
进入个人生态系统
  -> 看到当前生态状态
  -> 选择性记录一个观察点
  -> 本地保存
  -> 查看今日生态轨迹
  -> 查看生活晴雨表
```

用户可以只查看，不记录。系统不能在进入页面时强制弹出记录表，也不能把记录做成连续打卡压力。

同一次进入个人生态系统页面时，同一个维度最多保留一个新观察点。用户在同一页面会话里再次选择同一维度时，系统应覆盖刚刚记录的那一条，而不是不断追加挤满今日轨迹。用户返回启动面板并重新进入后，才可以为同一维度产生新的观察点。

### 3.2 语义观察点

v1.2 使用“语义观察点”作为个人生态系统的记录模型。

语义观察点是一条本地事件记录，表示用户在某个时刻对某个维度做了一次观察。

用户看到的是语义状态，例如：

- 疲惫。
- 可接受。
- 顺畅。
- 几乎没睡。

系统内部可以把语义状态映射为 `internalScore`，用于本地确定性统计。内部分数不作为用户侧评分展示。

### 3.3 六个生态维度

v1.2 第一版包含六个维度，顺序固定为：

1. 睡眠恢复。
2. 作息节律。
3. 身体状态。
4. 饮食饮水。
5. 活动舒展。
6. 环境支撑。

维度必须配置化。新增维度应优先通过配置完成，而不是复制一套业务逻辑。

### 3.4 三个生态面板

个人生态系统页面包含三个核心面板：

1. 当前生态状态。
2. 今日生态轨迹。
3. 生活晴雨表。

当前生态状态展示每个维度的最近观察。今日生态轨迹展示当天 0 点以后产生的观察点。生活晴雨表展示过去 1 天、7 天、15 天、30 天的本地观察摘要。

## 4. v1.2 修改内容

### 4.1 启动面板入口

v1.1 启动面板中六个子系统都是入口卡片。v1.2 将个人生态系统入口从“后续开放”推进为可进入状态。

要求：

- 点击个人生态系统进入个人生态系统页面。
- 个人生态系统被推荐时，推荐卡片也能进入同一页面。
- 其他五个子系统仍显示后续开放反馈。
- 从个人生态系统页面可以返回 LifeOS 启动面板。

### 4.2 本地数据

v1.2 在本地数据模块中新增生态观察点存储。

本地数据至少保存：

- onboarding 回答。
- 启动扫描结果。
- 个人生态系统观察点。

重置本地数据时，应同时清空个人生态系统观察点。

### 4.3 当前版本文档

v1.2 开发期间，`doc/current.md` 指向 v1.2。

v1.1 文档保持历史版本边界，不因 v1.2 新需求而改写。

## 5. v1.2 不包含内容

v1.2 不包含：

- 健康建议。
- 医疗建议。
- 心理诊断。
- 人格诊断。
- AI 分析。
- AI 推理。
- 账号系统。
- 云同步。
- 云端数据库。
- 服务器端保存个人生活数据。
- 用户直接编辑内部分数。
- 1 到 5 分或 1 到 10 分的显性评分 UI。
- 连续打卡、连续天数、惩罚或完成率压力。
- 复杂习惯系统。
- 家务清单系统。
- 其他五个子系统的真实业务闭环。
- 完整个人说明书。
- JSON 导出。
- Markdown 导出。
- Markdown 导入。
- 备份恢复。

## 6. 产品流程

v1.2 的首次使用路径继承 v1.1：

1. 用户在本地打开 LifeOS。
2. 如果没有本地扫描记录，展示启动界面和 onboarding。
3. 用户完成 9 道题。
4. LifeOS 保存扫描回答。
5. LifeOS 通过确定性规则生成初始扫描结果。
6. LifeOS 展示启动面板。
7. 启动面板展示六个子系统入口。

v1.2 新增的个人生态系统路径：

1. 用户点击个人生态系统入口。
2. LifeOS 展示个人生态系统页面。
3. 用户看到六个生态维度的当前状态。
4. 用户可以选择任一维度进行一次观察记录。
5. LifeOS 保存观察点到 IndexedDB。
6. 页面更新当前生态状态。
7. 今日生态轨迹显示当天观察点。
8. 生活晴雨表显示过去 1 天、7 天、15 天、30 天的观察摘要。
9. 用户可以返回 LifeOS 启动面板。

## 7. 界面设计

### 7.1 个人生态系统页面

页面主标题：

```text
个人生态系统
```

系统定义：

```text
生理基础与生活环境
```

核心区块：

1. 当前生态状态。
2. 今日生态轨迹。
3. 生活晴雨表。
4. 返回启动面板入口。

### 7.2 当前生态状态

当前生态状态展示六张维度卡片：

1. 睡眠恢复。
2. 作息节律。
3. 身体状态。
4. 饮食饮水。
5. 活动舒展。
6. 环境支撑。

每张卡片包含：

- 维度名称。
- 一句说明。
- 最近一次语义状态。
- 记录时间。
- 更新入口。

如果某个维度今天没有记录，应显示：

```text
今日未观察
```

如果某个维度从未记录，应显示：

```text
尚未观察
```

如果最近一次记录不是今天，可以展示最近记录日期，但不能暗示它代表今天。

### 7.3 记录观察点

用户点击某个维度的更新入口后，可以选择该维度的一个语义状态。

要求：

- 一次只更新一个维度。
- 不要求填写其他维度。
- 七个语义状态应呈现为一条从“支撑高”到“拖累高”的档位轴，而不是散落按钮。
- 可选备注可以保留到后续实现；如果 v1.2 实现备注，备注必须保存在本地。
- 保存后生成一个观察点。
- 同一次进入页面内，同一维度重复保存应覆盖本次进入刚生成的观察点。
- 用户可以删除误触生成的观察点。
- UI 不显示 `internalScore`。

### 7.4 今日生态轨迹

今日生态轨迹展示当天 0 点以后产生的观察点。

要求：

- 按维度分组。
- 每个观察点显示语义状态和时刻。
- 每个观察点提供删除入口，用于处理误触。
- 没有记录的时刻保持空白。
- 不对空白时段做插值。
- 不使用折线连接不连续的观察点。

“0 点以后清零”只表示今日视图换日，不表示删除历史数据。

### 7.5 生活晴雨表

生活晴雨表支持范围切换：

- 1 天。
- 7 天。
- 15 天。
- 30 天。

每个范围按维度展示：

- 观察次数。
- 平均语义状态。
- 低位观察次数。
- 最近一次观察。
- 一句确定性摘要。

摘要必须使用“观察”“记录”“线索”一类语言，不能使用“诊断”“建议”“你应该”一类语言。

## 8. 生态维度定义

### 8.1 内部状态

每个维度使用 7 档内部状态：

```text
+3 很支撑
+2 支撑
+1 略微支撑
 0 中性或可用
-1 略微拖累
-2 拖累
-3 明显拖累
```

`没观察` 不是 7 档之一，也不应写入平均值。没有观察就是没有数据。

### 8.2 睡眠恢复

| 内部状态 | 用户语义 |
| --- | --- |
| +3 | 很恢复 |
| +2 | 够用 |
| +1 | 勉强够 |
| 0 | 断续 |
| -1 | 偏少 |
| -2 | 很差 |
| -3 | 几乎没睡 |

### 8.3 作息节律

| 内部状态 | 用户语义 |
| --- | --- |
| +3 | 顺畅 |
| +2 | 稳定 |
| +1 | 基本在轨 |
| 0 | 有点散 |
| -1 | 被打断 |
| -2 | 很混乱 |
| -3 | 昼夜反转 |

### 8.4 身体状态

| 内部状态 | 用户语义 |
| --- | --- |
| +3 | 轻盈 |
| +2 | 稳定 |
| +1 | 可用 |
| 0 | 有点沉 |
| -1 | 疲惫 |
| -2 | 紧绷 |
| -3 | 不适 |

### 8.5 饮食饮水

| 内部状态 | 用户语义 |
| --- | --- |
| +3 | 很稳定 |
| +2 | 基本稳定 |
| +1 | 还可以 |
| 0 | 有点乱 |

| -2 | 过量或失衡 |
| -3 | 忘了照顾 |

### 8.6 活动舒展

| 内部状态 | 用户语义 |
| --- | --- |
| +3 | 很舒展 |
| +2 | 有活动 |
| +1 | 走动过 |

## 9. 数据模型草案

### 9.1 生态维度 ID

```ts
type EcosystemDimensionId =
  | "sleepRecovery"
  | "dailyRhythm"
  | "bodyState"
  | "foodWater"
  | "activityStretch"
  | "environmentSupport";
```

### 9.2 语义值

```ts
type EcosystemInternalScore = -3 | -2 | -1 | 0 | 1 | 2 | 3;

type EcosystemSemanticValue = {
  id: string;
  label: string;
  internalScore: EcosystemInternalScore;
};
```

### 9.3 维度定义

```ts
type EcosystemDimensionDefinition = {
  id: EcosystemDimensionId;
  label: string;
  description: string;
  values: EcosystemSemanticValue[];
};
```

### 9.4 观察点

```ts
type EcosystemObservation = {
  id: string;
  dimensionId: EcosystemDimensionId;
  valueId: string;
  valueLabel: string;
  internalScore: EcosystemInternalScore;
  observedAt: string;
  note?: string;
};
```

观察点必须保存 `valueLabel` 和 `internalScore` 快照，避免未来调整维度配置后重新解释历史记录。

### 9.5 生活晴雨表摘要

```ts
type EcosystemBarometerRange = "1d" | "7d" | "15d" | "30d";

type EcosystemDimensionSummary = {
  dimensionId: EcosystemDimensionId;
  observationCount: number;
  averageInternalScore: number | null;
  lowObservationCount: number;
  latestObservation: EcosystemObservation | null;
  summaryLabel: string | null;
};
```

## 10. 本地时间规则

今日生态轨迹按浏览器本地日期分组。

原则：

- 今日从用户设备本地时间 00:00 开始。
- 1 天、7 天、15 天、30 天统计同样使用本地日期边界。
- 0 点后今日轨迹自然为空，但历史数据不删除。
- 不引入服务器时间。

## 11. 聚合规则

### 11.1 当前生态状态

每个维度取最近一次观察点。

展示时区分：

- 今日已观察。
- 最近观察不是今天。
- 从未观察。

### 11.2 今日生态轨迹

筛选 `observedAt` 位于本地今天范围内的观察点。

按维度分组，再按时间升序展示。

### 11.3 生活晴雨表

对指定范围内的观察点按维度分组。

每个维度计算：

- `observationCount`
- `averageInternalScore`
- `lowObservationCount`
- `latestObservation`
- `summaryLabel`

`lowObservationCount` 统计 `internalScore <= -1` 的观察点。

`summaryLabel` 由平均值映射回该维度最接近的语义状态。它不是诊断，只是这段记录的平均位置。

## 12. 模块设计

### 12.1 App Shell 模块

职责：

- 继续负责首次启动、onboarding、启动面板和重置流程。
- 新增个人生态系统页面模式。
- 处理从启动面板进入个人生态系统。
- 处理从个人生态系统返回启动面板。

不应包含：

- 生态维度配置。
- 生态聚合规则。
- 生态观察点保存细节。

### 12.2 Startup Dashboard 模块

职责：

- 继续展示 LifeOS 启动面板。
- 个人生态系统入口可进入。
- 个人生态系统被推荐时，推荐卡片可进入个人生态系统。
- 其他五个子系统继续显示后续开放反馈。
- 个人生态系统的记录控件应保持复古 Macintosh 控件感，包括清晰黑色边框、凹凸阴影、条纹或点阵质感，以及可读的档位刻度。

不应重新引入：

- 完整个人说明书入口。
- JSON 导出入口。
- Markdown 导出入口。

### 12.3 Personal Ecosystem 模块

职责：

- 定义生态维度配置。
- 展示个人生态系统页面。
- 展示当前生态状态。
- 允许用户选择性记录某一维度观察点。
- 展示今日生态轨迹。
- 展示生活晴雨表。

不应包含：

- onboarding 题目。
- 子系统推荐规则。
- 账号、云端或 AI 逻辑。

### 12.4 Ecosystem Insights 模块

职责：

- 根据观察点计算当前状态。
- 根据观察点计算今日轨迹。
- 根据观察点计算生活晴雨表。
- 所有逻辑必须本地确定性。

不应包含：

- React UI。
- IndexedDB 读写。
- AI 分析或诊断文案。

### 12.5 Local Data 模块

职责：

- 保存 onboarding 回答。
- 保存启动扫描结果。
- 保存个人生态系统观察点。
- 按时间范围读取观察点。
- 清空所有本地 LifeOS 数据。

不应作为用户侧能力提供：

- JSON 导出。
- Markdown 导出。

## 13. 测试策略

v1.2 需要覆盖：

- 生态维度配置测试。
- 生态观察点本地存储测试。
- 当前生态状态计算测试。
- 今日生态轨迹计算测试。
- 生活晴雨表聚合测试。
- 个人生态系统页面组件测试。
- 启动面板入口测试。
- App Shell 进入和返回个人生态系统测试。
- 重置清空生态观察点测试。

关键验收测试：

1. 新用户仍然可以完成首次启动扫描。
2. 完成扫描后仍然进入 LifeOS 启动面板。
3. 启动面板仍然展示六个子系统。
4. 点击个人生态系统可以进入个人生态系统页面。
5. 其他五个子系统仍显示后续开放反馈。
6. 个人生态系统展示六个维度。
7. 用户可以只记录一个维度。
8. 记录后当前生态状态更新。
9. 今日生态轨迹展示当天观察点。
10. 生活晴雨表支持 1 天、7 天、15 天、30 天范围。
11. UI 不展示内部分数。
12. UI 不强制用户记录。
13. UI 没有连续打卡、完成率或惩罚文案。
14. 重置后清空 onboarding、启动扫描和生态观察点。
15. 不引入账号、云同步、云数据库或 AI 分析。

建议最终验证命令：

```bash
npm run test
npm run typecheck
npm run lint
npm run build
```

## 14. 已确认决策

以下决策适用于 v1.2：

1. v1.2 继续使用 Next.js + React + TypeScript。
2. 本地持久化继续使用 IndexedDB + Dexie。
3. 本地优先原则不变。
4. 六个子系统是 LifeOS 的长期主体。
5. v1.2 只开启个人生态系统的真实业务闭环。
6. 其他五个子系统仍保持后续开放。
7. 个人生态系统使用语义观察点模型。
8. 用户侧展示语义状态，不展示内部分数。
9. 生活晴雨表使用本地确定性聚合。
10. v1.2 不使用 AI 分析、诊断、账号、云同步或云端数据库。
