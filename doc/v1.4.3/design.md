# LifeOS v1.4.3 设计文档

## 1. 版本定位

LifeOS v1.4.3 是财务管理系统的第三个增量业务版本。

v1.4 已经开启财务管理系统，并完成“我的钱包”的本地余额快照闭环。v1.4.1 已经新增“收入来源”面板，完成“收入来源 -> 手动入账 -> 我的钱包更新”的闭环。v1.4.2 已经新增“日常开销池”面板，完成“钱包容器 -> 划入开销池 -> 消费扣减 -> 本地消费流水”的闭环。

v1.4.3 不修改 v1.4、v1.4.1 或 v1.4.2 的历史边界，不把 v1.4.3 的新需求写回 `doc/v1.4/`、`doc/v1.4.1/` 或 `doc/v1.4.2/`。

v1.4.3 在 v1.4.2 的基础上新增第四个真实财务面板：

> 财富流动日志。

v1.4.3 不是完整记账系统、完整账本事件源、预算系统、财务诊断系统或财务顾问。它只在现有业务动作之上新增一个本地、手动、追加式、可扩展的资金事件记录层，用于记录从 v1.4.3 起发生的关键财富流动，并为后续梦想账户和金鹅账户的财富流转记录预留结构。

v1.4.3 真正新增的可用业务只有：

- 财富流动日志。

## 2. 继承自 v1.4.2 的原则

v1.4.3 继续继承这些原则：

- 本地优先。
- 用户数据保存在用户自己的设备上。
- 不使用账号系统。
- 不使用云同步。
- 不使用云端数据库保存个人生活数据。
- 不使用 AI 分析、AI 推理、财务诊断、投资诊断、职业判断、心理诊断或医疗建议。
- 技术栈继续使用 Next.js + React + TypeScript。
- 本地持久化继续使用 IndexedDB + Dexie。
- 界面语言使用简体中文。
- 保留复古 Macintosh 风格和旧电脑感。
- onboarding 仍然只是初始扫描和子系统推荐器。
- 个人说明书仍然是后期板块，不是当前主功能。
- 推荐、扫描线索、子系统展示和财务计算都必须是确定性逻辑，不得伪装成诊断、建议或自动判断。

v1.4.3 继续保持这些已完成能力：

- 个人生态系统 v1.2 闭环。
- 能量管理系统 v1.3 闭环。
- 启动面板和六个子系统入口。
- 财务管理系统可进入。
- “我的钱包”本地余额快照闭环。
- “收入来源”本地手动入账闭环。
- “日常开销池”本地手动划款和消费扣减闭环。
- 重置本地数据时清空生态观察点、能量观察点、钱包容器、收入来源、日常开销池和消费流水。

v1.4.3 不继承 v1.4.2 中“完整财富流动日志不包含”的范围限制。v1.4.3 会把财富流动日志从后续开放结构推进为当前可用面板。

## 3. v1.4.3 新增内容

### 3.1 财富流动日志面板

v1.4.3 在财务管理系统页面新增“财富流动日志”面板。

财富流动日志用于展示 v1.4.3 之后新发生的关键资金流动事件。它不是完整记账系统，也不替代钱包容器、收入来源或日常开销池的现有状态模型。

财富流动日志只记录四类事件：

1. 入账。
2. 划款。
3. 消费。
4. 消费回退。

### 3.2 入账日志

用户对收入来源执行手动入账并成功后，系统生成一条财富流动日志。

固定金额收入来源：

- 使用收入来源的固定金额作为日志金额。
- 日志方向为 `in`。
- 日志类型为 `income_received`。
- 来源可以记录收入来源快照。
- 目标记录流入钱包容器快照。

不固定金额收入来源：

- 使用用户本次输入的到账金额作为日志金额。
- 日志方向为 `in`。
- 日志类型为 `income_received`。
- 来源可以记录收入来源快照。
- 目标记录流入钱包容器快照。

入账日志只记录 v1.4.3 之后发生的新入账。v1.4.1 或 v1.4.2 中已经发生但没有日志的数据不追溯、不补录。

### 3.3 划款日志

用户从钱包容器划入日常开销池并成功后，系统生成一条财富流动日志。

划款日志要求：

- 日志类型为 `daily_expense_transfer`。
- 日志方向为 `transfer`。
- 金额为本次划入金额。
- 来源记录钱包容器快照。
- 目标记录日常开销池快照。
- 发生时间使用划款确认成功时的本地时间。

划款日志不改变 v1.4.2 的日常开销池规则。划款仍然只保存开销池当前余额和最近一次划款信息；财富流动日志只是从 v1.4.3 起额外记录本次成功动作。

### 3.4 消费日志

用户在日常开销池中完成消费扣款后，系统生成一条财富流动日志。

消费日志要求：

- 日志类型为 `daily_expense_spent`。
- 日志方向为 `out`。
- 金额为本次消费金额。
- 来源记录日常开销池快照。
- 可以关联本次创建的 `DailyExpenseEntry`。
- 备注使用消费备注。
- 发生时间使用消费确认成功时的本地时间。

消费仍然只扣日常开销池余额，不修改钱包容器余额。

### 3.5 消费回退日志

用户删除消费流水并把对应金额加回开销池时，系统生成一条财富流动日志。

消费回退日志要求：

- 日志类型为 `daily_expense_refund`。
- 日志方向为 `in`。
- 金额为被删除消费流水的金额。
- 目标记录日常开销池快照。
- 必须通过 `relatedEventId` 或 `relatedDailyExpenseEntryId` 关联原消费。
- 发生时间使用删除消费流水成功时的本地时间。

消费回退不是删除或修改原消费日志，而是追加一条新的“消费回退”日志。

## 6. 产品流程

v1.4.3 的首次使用路径继承 v1.4.2：

1. 用户在本地打开 LifeOS。
2. 如果没有本地扫描记录，展示启动界面和 onboarding。
3. 用户完成 9 道题。
4. LifeOS 保存扫描回答。
5. LifeOS 通过确定性规则生成初始扫描结果。
6. LifeOS 展示启动面板。
7. 启动面板展示六个子系统入口。
8. 用户点击财务管理系统入口。
9. LifeOS 展示财务管理系统页面。

v1.4.3 新增的财富流动日志路径：

1. 用户进入财务管理系统。
2. 用户看到“我的钱包”“收入来源”“日常开销池”和“财富流动日志”。
3. 用户对收入来源执行手动入账。
4. 入账成功后，钱包容器余额增加，并生成一条入账日志。
5. 用户从钱包容器划入日常开销池。
6. 划款成功后，来源钱包容器余额减少，开销池余额增加，并生成一条划款日志。
7. 用户在日常开销池中输入消费金额和备注进行消费扣款。
8. 消费成功后，开销池余额减少，内部保存消费流水，并生成一条消费日志。
9. 用户删除可回退的消费流水。
10. 删除成功后，开销池余额加回，并生成一条消费回退日志。
11. 用户在“财富流动日志”面板查看 v1.4.3 之后发生的财富流动。

## 8. 数据模型草案

### 8.1 财富流动事件类型

```ts
type WealthFlowEventType =
  | "income_received"
  | "daily_expense_transfer"
  | "daily_expense_spent"
  | "daily_expense_refund";
```

### 8.2 财富流动方向

```ts
type WealthFlowDirection = "in" | "out" | "transfer";
```

说明：

- `in` 表示对用户当前可用资金或某个资金池的增加。
- `out` 表示对某个资金池的减少。
- `transfer` 表示系统内部资金位置变化。

### 8.3 财富流动主体

```ts
type WealthFlowSubjectType =
  | "wallet_container"
  | "income_source"
  | "daily_expense_pool"
  | "daily_expense_entry"
  | "dream_account"
  | "golden_goose_account";

type WealthFlowSubjectSnapshot = {
  type: WealthFlowSubjectType;
  id?: string;
  nameSnapshot: string;
};
```

说明：

- v1.4.3 实际使用 `wallet_container`、`income_source`、`daily_expense_pool` 和 `daily_expense_entry`。
- `dream_account` 和 `golden_goose_account` 只作为未来扩展预留，不实现对应业务。
- 日志保存名称快照，避免来源对象后续改名或删除后历史日志失去可读性。

### 8.4 财富流动日志

```ts
type WealthFlowEvent = {
  id: string;
  type: WealthFlowEventType;
  direction: WealthFlowDirection;
  amount: number;
  occurredAt: string;
  source?: WealthFlowSubjectSnapshot;
  target?: WealthFlowSubjectSnapshot;
  relatedEventId?: string;
  relatedDailyExpenseEntryId?: string;
  note?: string;
  createdAt: string;
  updatedAt: string;
};
```

说明：

- `amount` 必须为正数。
- 正负方向由 `direction` 和展示规则表达，不通过保存负数金额表达。
- `occurredAt` 是业务动作成功发生的本地时间。
- `source` 和 `target` 是事件发生时的快照。
- `relatedEventId` 用于关联消费回退和原消费日志。
- `relatedDailyExpenseEntryId` 用于关联消费回退和原消费流水。
- `note` 保存事件备注或用户输入备注快照。

## 9. 业务规则

### 9.1 入账日志规则

```text
用户对收入来源执行手动入账
  -> 校验收入来源和目标钱包容器可用
  -> 计算本次入账金额
  -> 更新目标钱包容器余额
  -> 创建 WealthFlowEvent(type = income_received)
  -> 保存钱包容器
  -> 保存财富流动日志
  -> 刷新我的钱包和财富流动日志
```

如果入账失败，不创建财富流动日志。

### 9.2 划款日志规则

```text
用户从钱包容器划入日常开销池
  -> 校验来源钱包容器可用
  -> 校验划款金额
  -> 更新来源钱包容器余额
  -> 更新日常开销池余额和最近一次划款信息
  -> 创建 WealthFlowEvent(type = daily_expense_transfer)
  -> 保存钱包容器
  -> 保存日常开销池
  -> 保存财富流动日志
  -> 刷新我的钱包、日常开销池和财富流动日志
```

如果划款失败，不创建财富流动日志。

### 9.3 消费日志规则

```text
用户在日常开销池中消费扣款
  -> 校验消费金额和备注
  -> 更新日常开销池余额
  -> 创建 DailyExpenseEntry
  -> 创建 WealthFlowEvent(type = daily_expense_spent)
  -> 保存日常开销池
  -> 保存消费流水
  -> 保存财富流动日志
  -> 刷新日常开销池和财富流动日志
```

如果消费失败，不创建财富流动日志。

### 9.4 消费回退日志规则

```text
用户删除某条消费流水
  -> 找到 DailyExpenseEntry
  -> 找到可关联的原消费日志
  -> 更新日常开销池余额
  -> 删除 DailyExpenseEntry
  -> 创建 WealthFlowEvent(type = daily_expense_refund)
  -> 保存日常开销池
  -> 删除消费流水
  -> 保存财富流动日志
  -> 刷新日常开销池和财富流动日志
```

消费回退不修改钱包容器余额。

消费回退不删除原消费日志。

### 9.5 日志一致性规则

财富流动日志是业务动作成功后的记录。实现时应避免出现“余额已更新但日志未记录”或“日志已记录但业务动作失败”的半完成状态。

如果当前实现无法在 IndexedDB 中提供严格事务，应至少保证：

- 业务动作失败时不保存日志。
- 日志保存失败时给出明确错误，不静默吞掉。
- 不用日志反向重算钱包或开销池余额。

- 钱包汇总计算。

- 保存财务管理系统财富流动日志。
- 读取财富流动日志。
- 删除或清空财富流动日志。
- 清空所有本地 LifeOS 数据。

不应作为用户侧能力提供：

- JSON 导出。

## 12. 已确认决策

以下决策适用于 v1.4.3：

1. v1.4.3 继续使用 Next.js + React + TypeScript。
2. 本地持久化继续使用 IndexedDB + Dexie。
3. 本地优先原则不变。
4. 六个子系统是 LifeOS 的长期主体。
5. v1.4.3 保持个人生态系统已完成闭环。
6. v1.4.3 保持能量管理系统已完成闭环。
7. v1.4.3 保持财务管理系统可进入。
8. v1.4.3 保持“我的钱包”余额快照闭环。
9. v1.4.3 保持“收入来源”手动入账闭环。
10. v1.4.3 保持“日常开销池”手动划款和消费扣减闭环。
11. v1.4.3 新增“财富流动日志”真实业务面板。
12. v1.4.3 的财富流动日志只记录 v1.4.3 之后发生的新事件。
13. v1.4.3 只记录入账、划款、消费和消费回退四类事件。
14. v1.4.3 消费回退通过追加日志表达，不删除或改写原消费日志。
15. v1.4.3 不把财富流动日志作为钱包或开销池的唯一事实来源。
16. v1.4.3 取消日常开销池用户侧消费流水可见板块。
17. v1.4.3 不做旧数据迁移、完整记账、预算统计、财务诊断、账号、云同步或云端数据库。
