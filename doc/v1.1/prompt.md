# LifeOS v1.1 Agent Prompt

## 目标

根据当前项目文档，实现 LifeOS v1.1。

v1.1 是一次减法重构版本。目标是把 v1.0 从“个人说明书原型”纠偏为“六个子系统驱动的本地人生操作系统启动器”。

实现过程应支持多 agent 协作。主 Agent 负责整体集成、任务分派、进度跟踪和最终验证；子 Agent 按 `doc/v1.1/tasks/` 中的任务文件领取模块。

## 输入

必须先阅读：

1. `doc/current.md`
2. `doc/README.md`
3. `doc/decisions/`
4. `doc/v1.1/design.md`
5. `doc/v1.1/tasks/progress.md`
6. `doc/v1.1/tasks/`

必要时参考：

- `doc/v1.0/design.md`
- `doc/v1.0/tasks/`
- `doc/roadmap/v1.1.md`

参考 v1.0 时，只继承仍然有效的本地优先、技术栈、模块边界和测试原则。不要复制 v1.0 中已经过时的产品范围。

## 输出

完成 LifeOS v1.1 的可运行实现。

v1.1 实现后应具备：

- 首次启动扫描流程。
- 9 道 onboarding 题。
- 新增个人生态系统。
- 六个子系统作为主界面主体。
- LifeOS 启动面板。
- 最多 2 个建议优先开启的子系统。
- 推荐来源展示。
- 扫描线索展示。
- 本地保存、读取和重置。

v1.1 实现后不应具备用户侧：

- 完整个人说明书编辑器。
- JSON 导出入口。
- Markdown 导出入口。
- 个人说明书系统推荐。
- 自我清晰度主区块。

## 技术要求

- 应用框架：Next.js。
- 前端框架：React。
- 开发语言：TypeScript。
- 本地数据：IndexedDB。
- IndexedDB 封装：Dexie。
- 测试：沿用当前项目测试方案。
- 包管理器：沿用当前项目配置。

必须通过：

```bash
npm run test
npm run typecheck
npm run lint
```

不要引入：

- 后端服务。
- 账号系统。
- 云同步。
- 云端数据库。
- 外部 AI 服务。
- 用于分析、推理或诊断用户的 AI 能力。

## 核心产品约束

LifeOS v1.1 必须保持本地优先：

- 用户数据只保存在用户自己的设备上。
- 默认不向任何云端上传个人生活数据。
- 不实现账号系统、云同步或多人协作。
- 不使用 AI 分析、AI 推理、心理诊断或人格诊断。
- 推荐和扫描线索由确定性规则生成。
- 推荐和扫描线索必须保留来源回答。

界面语言使用简体中文。

## 模块边界

严格按照 `doc/v1.1/design.md` 和 `doc/v1.1/tasks/` 中的模块职责实现。

### App Shell

负责首次启动判断和高层流程切换：

- 无本地扫描结果时展示 onboarding。
- 有本地扫描结果时展示 LifeOS 启动面板。
- 重置后回到首次启动状态。

不包含问题内容、推荐规则或子系统业务逻辑。

### Onboarding

负责 9 道题的首次启动扫描：

- 第 1-8 题为多选题。
- 每道多选题最少选择 1 个。
- 不设置最大选择数量。
- 第 1-8 题支持“其他”文本回答。
- 第 9 题可跳过。
- 完成前不写入持久化数据。

### Question Schema

作为 9 道题的静态事实来源。

要求：

- 定义问题、选项、校验规则、signal tags 和 write targets。
- 为六个子系统推荐提供信号。
- 包含个人生态系统相关信号。
- 不再包含 `子系统:manual`。

### Subsystem Recommendation

负责根据扫描回答推荐子系统。

候选子系统只能包括：

- 个人生态系统。
- 能量管理系统。
- 认知管理系统。
- 人生目标管理系统。
- 人际关系管理系统。
- 财务管理系统。

不再推荐个人说明书系统。

### Startup Scan Generation

负责把 onboarding 回答转换为启动扫描结果。

要求：

- 生成扫描线索。
- 生成最多 2 个建议子系统。
- 每条线索和推荐保留来源回答。
- 不生成可编辑个人说明书章节。
- 不生成自我清晰度主概念。

### Startup Dashboard

负责展示 LifeOS 启动面板。

要求：

- 页面标题为 `LifeOS 启动面板`。
- 展示六个子系统。
- 展示推荐系统和推荐来源。
- 展示扫描线索。
- 提供重置入口。
- 不展示 JSON / Markdown 导出入口。
- 不展示完整个人说明书编辑入口。

### Local Data

负责本地保存、读取和重置。

要求：

- 保存 onboarding 回答。
- 保存启动扫描结果。
- 读取本地扫描结果。
- 清空本地 LifeOS 数据。
- 不作为用户侧能力提供 JSON 或 Markdown 导出。

## Agent 工作方式

主 Agent 负责：

- 阅读 v1.1 全部输入文档。
- 检查当前代码与 v1.1 目标的差异。
- 按 `doc/v1.1/tasks/` 分派子 Agent。
- 确保并行任务不会互相覆盖。
- 审核子 Agent 输出。
- 集成模块。
- 更新 `doc/v1.1/tasks/progress.md`。
- 运行最终验证。

子 Agent 负责：

- 只处理领取的任务文件。
- 补充或更新对应测试。
- 保持模块边界清晰。
- 不修改无关模块。
- 不引入云端依赖。
- 不重复定义核心类型。
- 不把个人说明书、导入导出或 AI 能力重新引入 v1.1。

## 并行开发建议

推荐顺序：

1. 先完成子系统模型与推荐规则。
2. 同步修正 onboarding schema。
3. 再重构启动扫描生成和本地数据。
4. 再重构 LifeOS 启动面板和删除用户侧导出。
5. 最后做 App Shell 集成、文档同步和全量验收。

## 完成标准

v1.1 完成时必须满足：

- `doc/v1.1/tasks/progress.md` 所有任务完成。
- 所有测试通过。
- TypeScript 类型检查通过。
- lint 通过。
- 本地页面可完成首次扫描。
- 完成扫描后进入 LifeOS 启动面板。
- 六个子系统全部出现。
- 个人生态系统可被推荐。
- 个人说明书系统不再被推荐。
- 用户界面没有 JSON 导出。
- 用户界面没有 Markdown 导出。
- 用户界面没有完整个人说明书编辑入口。
- 重置后可以回到首次启动状态。
