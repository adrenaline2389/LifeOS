# LifeOS v1.1 设计文档

## 1. 版本定位

LifeOS v1.1 是一次减法重构版本。

v1.0 验证了本地优先、首次引导、确定性规则生成、本地持久化、编辑、导出和重置等基础能力。但 v1.0 的产品重心偏向“个人说明书控制面板”，容易让 LifeOS 看起来像一份 onboarding 后生成的说明书。

v1.1 的目标是把 LifeOS 纠偏为“六个子系统驱动的本地人生操作系统启动器”。

v1.1 完成后，用户完成首次启动扫描后看到的重点不再是个人说明书，而是：

- 六个子系统地图。
- 最多 2 个建议优先开启的系统。
- 推荐来源和扫描线索。
- 重置本地数据入口。

## 2. 继承自 v1.0 的原则

v1.1 继续继承这些原则：

- 本地优先。
- 用户数据保存在用户自己的设备上。
- 不使用账号系统。
- 不使用云同步。
- 不使用云端数据库保存个人生活数据。
- 不使用 AI 分析、AI 推理、心理诊断或人格诊断。
- 使用确定性规则处理 onboarding 回答。
- 推荐和扫描线索必须保留来源回答。
- 界面语言使用简体中文。
- 技术栈继续使用 Next.js + React + TypeScript。
- 本地持久化继续使用 IndexedDB + Dexie。
- 保留复古 Macintosh 风格和旧电脑感。

v1.1 不继承 v1.0 中已经过时的范围：

- 不继续把个人说明书作为当前主功能。
- 不继续提供完整个人说明书编辑器。
- 不继续提供用户侧 JSON 导出。
- 不继续提供用户侧 Markdown 导出。
- 不继续推荐“个人说明书系统”。
- 不继续把“自我清晰度”作为主界面核心概念。

## 3. v1.1 新增内容

### 3.1 新增个人生态系统

v1.1 正式新增第六个顶层子系统：

> 个人生态系统：生理基础与生活环境。

个人生态系统关注：

- 作息。
- 睡眠。
- 饮食。
- 饮水。
- 身体状态。
- 运动。
- 居住环境。
- 日常维护。
- 生活节律。

它回答的问题是：

> 我的身体和环境能不能支撑我运行？

### 3.2 六个子系统成为主界面主体

v1.1 的 LifeOS 启动面板展示六个子系统：

1. 个人生态系统：生理基础与生活环境。
2. 能量管理系统：心理余量与恢复。
3. 认知管理系统：信息、学习、判断和反思。
4. 人生目标管理系统：方向、项目和行动。
5. 人际关系管理系统：连接、沟通和边界。
6. 财务管理系统：资源、消费和自由度。

每个子系统在 v1.1 中都是入口卡片，不实现真实业务闭环。

### 3.3 初始扫描结果

v1.1 将 onboarding 结果称为“初始扫描结果”，而不是“个人说明书初稿”。

初始扫描结果包含：

- 扫描完成状态。
- 扫描线索。
- 最多 2 个建议优先开启的子系统。
- 推荐来源回答。

## 4. v1.1 修改内容

### 4.1 Onboarding 重新定位

9 道 onboarding 题保留，但定位从“生成个人说明书”改为“初始扫描和子系统推荐”。

题目仍遵守：

- 第 1-8 题为多选题。
- 每道多选题最少选择 1 个。
- 不设置最大选择数量。
- 第 1-8 题支持“其他”文本回答。
- 第 9 题为可跳过短文本题。
- 完成前不写入持久化数据。

### 4.2 推荐规则重新定位

推荐系统只推荐六个正式子系统：

- `ecosystem`
- `energy`
- `cognition`
- `goals`
- `relationships`
- `finance`

不再推荐：

- `manual`
- 个人说明书系统

推荐结果最多 2 个，每个推荐必须包含原因文案和来源回答。

### 4.3 个人说明书降级为后期板块

个人说明书在 v1.1 中不作为主功能出现。

未来的个人说明书应由六个子系统的长期使用情况沉淀生成。导入、导出和恢复能力也应围绕成熟的 Markdown 说明书重新设计。

## 5. v1.1 不包含内容

v1.1 不包含：

- 完整个人说明书。
- 个人说明书编辑器。
- JSON 导出。
- Markdown 导出。
- Markdown 导入。
- 备份恢复。
- 真实子系统业务数据。
- 账号系统。
- 云同步。
- 云端数据库。
- 多人协作。
- AI 分析。
- AI 心理诊断。
- 人格诊断。

## 6. 产品流程

v1.1 的首次使用路径：

1. 用户在本地打开 LifeOS。
2. 如果没有本地扫描记录，展示启动界面和 onboarding。
3. 用户完成 9 道题。
4. LifeOS 保存扫描回答。
5. LifeOS 通过确定性规则生成初始扫描结果。
6. LifeOS 展示启动面板。
7. 启动面板展示六个子系统入口。
8. 推荐的 1-2 个子系统显示“建议优先开启”。
9. 用户可以查看推荐来源。
10. 用户可以重置本地数据并重新扫描。

## 7. 界面设计

### 7.1 启动面板

页面主标题：

```text
LifeOS 启动面板
```

启动状态：

```text
初始扫描完成
```

核心区块：

1. 启动状态。
2. 建议优先开启的系统。
3. 六个子系统地图。
4. 扫描线索。
5. 重置本地数据。

不再展示：

- 自我清晰度。
- 完整个人说明书入口。
- 说明书章节编辑器。
- JSON 导出按钮。
- Markdown 导出按钮。

### 7.2 子系统卡片

每个子系统卡片包含：

- 系统名称。
- 一句定义。
- 状态标签。
- 如果被推荐，展示“建议优先开启”。
- 如果未推荐，展示“未开启”或“后续开放”。

### 7.3 推荐来源

每个推荐必须展示来源回答。来源展示可以继续复用 v1.0 的 `SourceReference` 组件。

## 8. 数据模型草案

### 8.1 子系统 ID

```ts
type SubsystemId =
  | "ecosystem"
  | "energy"
  | "cognition"
  | "goals"
  | "relationships"
  | "finance";
```

### 8.2 初始扫描结果

```ts
type StartupScanProfile = {
  version: "1.1";
  completedAt: string;
  scanStatus: "completed";
  scanClues: StartupScanClue[];
  suggestedSubsystems: SuggestedSubsystem[];
};
```

### 8.3 扫描线索

```ts
type StartupScanClue = {
  id: string;
  text: string;
  sourceAnswerRefs: SourceAnswerRef[];
};
```

### 8.4 建议子系统

```ts
type SuggestedSubsystem = {
  id: SubsystemId;
  label: string;
  reason: string;
  sourceAnswerRefs: SourceAnswerRef[];
};
```

## 9. 模块设计

### 9.1 App Shell 模块

职责：

- 判断是否已有本地扫描结果。
- 无扫描结果时展示 onboarding。
- 有扫描结果时展示 LifeOS 启动面板。
- 重置后回到 onboarding。

不应包含：

- 问题内容。
- 推荐规则。
- 子系统业务逻辑。

### 9.2 Onboarding 模块

职责：

- 渲染 9 道首次启动扫描题。
- 校验回答规则。
- 产出结构化回答记录。

不应包含：

- 推荐规则。
- 扫描结果生成逻辑。

### 9.3 问题 Schema 模块

职责：

- 作为 9 道题的静态事实来源。
- 定义选项、signal tags 和 write targets。
- 为六个子系统推荐提供输入信号。

### 9.4 子系统推荐模块

职责：

- 定义六个候选子系统。
- 根据回答信号给子系统计分。
- 返回最多 2 个推荐。
- 附加推荐原因和来源回答。

### 9.5 初始扫描生成模块

职责：

- 将 onboarding 回答转换为启动扫描结果。
- 生成扫描线索。
- 调用子系统推荐模块。
- 不生成个人说明书章节。
- 不生成自我清晰度主概念。

### 9.6 LifeOS 启动面板模块

职责：

- 展示启动状态。
- 展示六个子系统。
- 展示建议优先开启的系统和来源。
- 展示扫描线索。
- 提供重置入口。

### 9.7 本地数据模块

职责：

- 保存 onboarding 回答。
- 保存启动扫描结果。
- 读取本地数据。
- 清空本地数据。

不再作为用户侧能力提供：

- JSON 导出。
- Markdown 导出。

### 9.8 重置模块

职责：

- 提供明确确认后的本地数据重置能力。
- 未确认时不执行重置。
- 重置后通知 App Shell 回到首次启动状态。

## 10. 测试策略

v1.1 需要覆盖：

- 问题 schema 测试。
- onboarding 校验测试。
- 子系统推荐测试。
- 初始扫描生成测试。
- 本地数据保存、读取、重置测试。
- 启动面板组件测试。
- App Shell 首次启动、已完成扫描、重置流程测试。

关键验收测试：

1. 新用户可以完成 9 道题。
2. 完成后进入 LifeOS 启动面板。
3. 启动面板展示六个子系统。
4. 个人生态系统出现在六个子系统中。
5. 推荐结果最多 2 个。
6. 个人生态系统可以被推荐。
7. 个人说明书系统不再被推荐。
8. 界面没有 JSON 导出入口。
9. 界面没有 Markdown 导出入口。
10. 界面没有完整个人说明书编辑入口。
11. 重置后回到首次启动状态。
12. 不引入云端依赖或 AI 分析。

## 11. 已确认决策

以下决策适用于 v1.1：

1. v1.1 继续使用 Next.js + React + TypeScript。
2. 本地持久化继续使用 IndexedDB + Dexie。
3. 本地优先原则不变。
4. 六个子系统是 LifeOS 的长期主体。
5. 个人生态系统是第六个顶层子系统。
6. Onboarding 是初始扫描和推荐器，不是核心产品。
7. 个人说明书是未来后期板块，不是 v1.1 主功能。
8. v1.1 删除用户侧 JSON 和 Markdown 导出。
9. v1.1 不使用 AI 分析或心理诊断。
