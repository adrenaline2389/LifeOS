# 0003: 六个子系统是 LifeOS 的长期主体

## 状态

Accepted

## 背景

v1.0 通过 9 道 onboarding 问题生成初始个人说明书控制面板。这个版本验证了 LifeOS 可以作为本地优先的 Web App 运行，也验证了首次引导、确定性规则生成、用户编辑、本地导出和重置等基础能力。

但 v1.0 的产品叙事容易让 LifeOS 看起来像一个围绕 onboarding 和个人说明书展开的应用。这个重心不符合 LifeOS 的长期方向。

LifeOS 的目标不是把 9 道题生成的早期说明书持续打磨成核心产品，而是逐步搭建一套本地运行的人生操作系统。它的主体应落在用户日常使用的子系统中。

## 决策

LifeOS 的长期主体是六个子系统：

1. 个人生态系统：管理生理基础与生活环境。
2. 能量管理系统：管理心理余量、压力信号和恢复方式。
3. 认知管理系统：管理信息、学习、判断和反思。
4. 人生目标管理系统：管理方向、项目和行动。
5. 人际关系管理系统：管理连接、沟通和边界。
6. 财务管理系统：管理资源、消费和自由度。

首次启动 onboarding 是辅助层。它的作用是初始扫描和推荐用户可以优先开启的子系统，而不是 LifeOS 的长期主体。

个人说明书也是辅助板块，但它应在后期出现。它不是第七个系统，而是由六个子系统的长期使用情况沉淀出的一份说明书。未来导入和导出能力应围绕这份说明书设计，优先考虑 Markdown 文件形态，让用户可以像恢复备份一样通过说明书重建六个系统的关键状态。

## 原因

这个决策让 LifeOS 的产品重心更稳定：

- 六个子系统覆盖一个人的身体、心理、认知、行动、关系和资源。
- onboarding 可以降低首次进入成本，但不应喧宾夺主。
- 个人说明书需要真实使用数据支撑，过早成为核心会让系统围绕“开胃前菜”打转。
- 导入导出能力应服务成熟的个人说明书，而不是过早围绕初始问卷结果设计。
- 后续版本可以围绕子系统逐步扩展，不必推翻 v1.0 的技术验证。

## 后果

正面影响：

- 后续版本的规划会优先考虑六个子系统的入口、数据模型和使用闭环。
- v1.0 可以被重新理解为早期启动原型，而不是错误方向。
- README、current 文档和未来版本文档可以采用更清晰的产品叙事。
- 个人生态系统被正式纳入 LifeOS 顶层结构，和能量管理系统形成“体力 / 心力”的底层分工。

代价：

- v1.0 中围绕个人说明书的部分需要在后续版本中降级或重新定位。
- 已实现的 JSON / Markdown 导出能力可能需要在后续版本中重新设计为围绕成熟个人说明书的备份和恢复能力。
- 后续开发需要避免继续把 onboarding、说明书和导出功能当作主线。

## 约束

后续版本规划应遵守：

- 不把 9 道 onboarding 问题扩展成主要产品体验。
- 不把早期个人说明书控制面板视为长期首页。
- 不把个人说明书设计成第七个系统。
- 不在六个子系统尚未建立前过度投入导入、导出和恢复能力。
- 优先建立六个子系统中的真实使用闭环。
