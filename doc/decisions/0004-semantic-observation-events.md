# 0004: 使用语义观察点作为子系统记录模型

## 状态

Accepted

## 背景

v1.1 将 LifeOS 的长期主体明确为六个子系统，并把当前产品形态收敛为六系统启动器。v1.2 计划开启第一个真实子系统闭环：个人生态系统。

个人生态系统管理的是作息、睡眠、身体状态、饮食饮水、活动和生活环境。这类数据很敏感，但又不适合通过强制打卡、健康评分或诊断方式呈现。

用户可能一天内多次进入个人生态系统，也可能只是查看状态而不记录。如果系统每天只允许记录一次，会丢失一天内的变化；如果每次进入都要求填写整张表，会像开屏广告一样打扰用户。

同时，LifeOS 未来还会陆续开启其他子系统。v1.2 需要一个可扩展的数据模型，让新增维度可以通过配置完成，而不是为每个维度复制业务逻辑。

## 决策

LifeOS v1.2 起，子系统中的轻量状态记录优先使用“语义观察点”模型。

语义观察点是一条本地事件记录，表示用户在某个时刻对某个维度做了一次观察。

以个人生态系统为例，一个观察点包含：

- 子系统 ID。
- 维度 ID。
- 语义值 ID。
- 语义标签快照。
- 内部状态分数。
- 观察时间。
- 可选备注。

用户侧选择语义选项，例如“疲惫”“可接受”“顺畅”。系统内部可以把语义值映射为 -3 到 +3 的 `internalScore`，用于本地确定性聚合。

内部分数不作为主要用户界面呈现。用户界面应展示语义状态、观察次数、范围摘要和低位线索，而不是把用户状态包装成显性评分。

没有观察不是一个分数。`没观察` 不进入平均值，也不应作为 0 分处理。

聚合规则必须是本地确定性的：

- 不使用 AI 分析。
- 不生成诊断。
- 不对稀疏数据做插值。
- 不暗示系统知道未记录时段的真实状态。
- 使用用户设备的本地日期边界。
- 0 点后今日视图自然换日，但历史观察点不删除。

本地持久化继续使用 IndexedDB + Dexie。不得把个人观察点上传到云端服务。

## 原因

这个模型适合 LifeOS 的产品气质：

- 用户可以选择性记录某个维度，不必填写整张表。
- 一天内可以记录多次，保留状态变化。
- 用户只看到语义，不被数字评分审判。
- 系统仍能通过内部数值做确定性统计。
- 稀疏数据被当作稀疏数据处理，不伪造连续趋势。
- 维度配置化后，后续新增维度成本较低。
- 该模型可以扩展到未来其他子系统的轻量状态记录。

## 后果

正面影响：

- v1.2 可以建立“记录 -> 本地保存 -> 回顾”的真实使用闭环。
- 用户进入个人生态系统时不会被强制打扰。
- 生活晴雨表可以基于本地记录生成，不需要 AI。
- 未来新增维度可以优先通过配置完成。
- 历史记录保存语义快照，降低后续配置变动对旧数据的影响。

代价：

- 数据模型比“每日一条记录”复杂。
- UI 需要清楚地区分今日未观察、最近观察和从未观察。
- 聚合逻辑需要处理稀疏数据和不同时间范围。
- 内部分数存在被误用为用户评分的风险，需要在界面和文案上持续约束。

## 约束

实现时应遵守：

- 用户界面不以 1 到 5 分、1 到 10 分或类似数字评分为主。
- 不做 streak、连续打卡、完成率惩罚或健康评分。
- 不把平均值描述为诊断、建议或事实结论。
- 不把缺失观察自动补成中间状态。
- 不用折线连接不连续的观察点来暗示连续变化。
- 不引入账号、云同步、云端数据库或 AI 分析。
- 新增云端能力、AI 分析或远程备份前，必须新增独立 ADR。
