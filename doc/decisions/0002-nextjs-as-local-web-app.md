# 0002: 使用 Next.js 作为本地 Web App 框架

## 状态

Accepted

## 背景

LifeOS v1.0 是一个网页端本地应用。它需要一个成熟的前端工程骨架来组织页面、组件、模块、样式和构建流程。

项目需要长期演进，未来可能会出现更多页面、子系统、导出视图、编辑界面和文档页面。因此，框架选择不仅要服务 v1.0，也要为后续版本留下清晰扩展空间。

## 决策

LifeOS 使用 Next.js + React + TypeScript 作为应用技术栈。

在 v1.0 中，Next.js 只作为本地 Web App 框架使用。

具体约束：

- 不使用 Next.js API Routes 保存、上传或分析个人生活数据。
- 不使用 Server Actions 处理个人生活数据。
- 不把用户个人数据写入服务端数据库。
- 与 onboarding、个人说明书、导入导出和本地持久化相关的逻辑应运行在浏览器端。
- IndexedDB 和 Dexie 是 v1.0 的主要持久化方案。

## 原因

选择 Next.js 的原因：

- 提供成熟的 React 应用组织方式。
- 便于后续扩展页面、路由和模块。
- TypeScript 支持良好，适合维护数据模型和模块边界。
- 社区生态成熟，适合开源协作。
- 即使 v1.0 不使用服务端能力，Next.js 仍可作为稳定的本地 Web App 骨架。

这个选择不表示 LifeOS v1.0 是云端应用，也不表示个人生活数据会进入服务端。

## 后果

正面影响：

- 项目结构更适合长期维护。
- 后续版本可以自然扩展更多页面和子系统。
- React 组件、测试和构建生态更成熟。
- 开源协作者更容易理解和参与。

代价：

- 相比更轻量的纯 Vite 应用，Next.js 框架心智更重。
- 需要持续约束服务端能力的使用，避免误把个人数据处理搬到服务端。
- 涉及 IndexedDB 的模块必须注意只在浏览器端运行。

## 实现准则

开发时应遵守：

- 个人数据相关模块使用客户端组件或浏览器端工具函数。
- 任何访问 `window`、`indexedDB`、下载文件或浏览器存储的逻辑，都不能在服务端渲染阶段直接执行。
- 如果使用 Next.js 服务端能力，只能用于框架默认构建、静态资源、路由组织或未来明确批准的非个人数据场景。
- 新增涉及服务端能力的功能前，必须确认是否违反 `0001-local-first.md`。
