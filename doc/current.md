## v1.4 状态

v1.4 是 LifeOS 的第三个真实子系统版本，目标是在 v1.3 的个人生态系统和能量管理系统闭环基础上，开启“财务管理系统”的基础设施版本。

v1.4 的核心方向：

- 保持本地优先。
- 继承 v1.3 的首次启动扫描、启动面板、六个子系统地图、个人生态系统闭环和能量管理系统闭环。
- 将“财务管理系统”从后续开放入口推进为可进入状态。
- 财务管理系统真正可用的业务只有“我的钱包”。
- 用户可以新建、编辑和删除资金容器。
- 用户直接编辑资金容器当前余额。
- 当前总余额根据资金容器自动汇总。
- 总余额为正时展示彩色环形分布。
- 总余额为零或负时展示灰色状态，不展示占比。
- 总余额为负时显示当前负债金额。
- 不做资金容器分类、记账流水、梦想账户、金鹅账户、金钱流入来源、日常开销池或财富流动日志业务。
- 不使用财务诊断、投资建议、预算警报、超支惩罚、AI 分析、账号系统、云同步、云端数据库、银行连接或支付连接。
- 不恢复用户侧 JSON 导出、Markdown 导出或完整个人说明书编辑入口。

## v1.3 状态

v1.3 是 LifeOS 的第二个真实子系统闭环版本，目标是在 v1.2 的个人生态系统闭环基础上，开启“能量管理系统”的最小观察闭环。

v1.3 的核心方向：

- 保持本地优先。
- 继承 v1.2 的首次启动扫描、启动面板、六个子系统地图和个人生态系统闭环。
- 将“能量管理系统”从后续开放入口推进为可使用的真实子系统。
- 使用语义观察点记录当前余量、压力负载、情绪天气、注意带宽、社交电量和行动阻力。
- 用户可以选择性记录某一个维度，不强制每天填写整张表。
- 展示当前能量状态、今日能量轨迹和能量风向标。
- 能量风向标支持 1 天、7 天、15 天、30 天范围。
- 用户侧展示语义状态，不展示内部分数。
- 不使用心理诊断、情绪诊断、健康建议、医疗建议、AI 分析、账号系统、云同步或云端数据库。
- 不引入任务规划、日程安排、恢复方案推荐、恢复方式库、跨系统推断或合并评分。
- 不恢复用户侧 JSON 导出、Markdown 导出或完整个人说明书编辑入口。

## v1.1 状态

v1.1 是 LifeOS 的减法重构版本，目标是把 v1.0 从“个人说明书原型”纠偏为“六个子系统驱动的本地人生操作系统启动器”。

v1.1 的核心方向：

- 保持本地优先。
- 保留 9 道 onboarding 题，但重新定位为首次启动扫描和子系统推荐器。
- 新增“个人生态系统”作为第六个顶层子系统。
- 完成 onboarding 后展示 LifeOS 启动面板。
- 启动面板展示六个子系统入口。
- 最多推荐 2 个建议优先开启的子系统。
- 推荐和扫描线索保留来源回答。
- 删除用户侧 JSON 导出。
- 删除用户侧 Markdown 导出。
- 删除完整个人说明书编辑入口。
- 不使用账号系统、云同步、云端数据库或 AI 分析。

## v1.0 状态

v1.0 是 LifeOS 的第一个本地可运行版本，目标是通过 9 道 onboarding 问题生成初始个人说明书控制面板。

从后续产品方向看，v1.0 应被理解为早期启动原型：它验证了本地优先、首次引导、确定性规则生成、编辑、导出和重置等基础能力。它不代表 LifeOS 的长期产品重心。

LifeOS 的长期主体应落在六个子系统上：

- 个人生态系统：生理基础与生活环境。
- 能量管理系统：心理余量与恢复。
- 认知管理系统：信息、学习、判断和反思。
- 人生目标管理系统：方向、项目和行动。
- 人际关系管理系统：连接、沟通和边界。
- 财务管理系统：资源、消费和自由度。

首次启动的 9 道 onboarding 问题只作为初始扫描和子系统推荐器。个人说明书应作为后期独立板块，由六个子系统的长期使用情况沉淀生成，而不是 LifeOS 的长期主体。

v1.0 的核心边界：

- 本地优先。
- 不使用账号系统。
- 不使用云同步。
- 不使用云端数据库保存个人生活数据。
- 不使用 AI 分析、AI 推理或心理诊断。
- 以确定性规则生成个人说明书初稿。
- 所有观察都标记为待验证观察。

## 工作规则

开发当前版本时，以 `doc/v1.4/design.md` 和 `doc/v1.4/tasks/` 为准。

如需理解历史实现背景，可以参考 `doc/v1.3/design.md`、`doc/v1.3/tasks/`、`doc/v1.2/design.md`、`doc/v1.2/tasks/`、`doc/v1.1/design.md`、`doc/v1.1/tasks/`、`doc/v1.0/design.md` 和 `doc/v1.0/tasks/`，但不要把历史版本中已经过时的产品范围带入 v1.4。

如果发现设计文档、任务文件和代码实现不一致：

1. 先判断是否属于文档错误、实现偏差或需求变化。
2. 当前版本文档错误可以在 v1.4 文档中修正。
3. 实现偏差应优先修代码，使其回到 v1.4 边界内。
4. 需求变化不要直接写进 v1.4，应先进入 `doc/roadmap/` 或新的版本目录。

## 后续版本

每个正式版本应新建独立目录，例如：

```text
doc/v2.0/
```

未来版本可以引用 v1.0 的设计，但不应直接覆盖 v1.0 的历史文档。

在规划后续版本时，应优先纠正 v1.0 的产品重心偏移：不要继续围绕 onboarding 或早期说明书扩展细节，而应逐步建立六个子系统的主界面、数据模型和真实使用闭环。
