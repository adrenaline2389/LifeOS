## v1.4.3 状态

v1.4.3 是财务管理系统的第三个增量业务版本，目标是在 v1.4 的“我的钱包”本地余额快照闭环、v1.4.1 的“收入来源”手动入账闭环和 v1.4.2 的“日常开销池”手动划款与消费扣减闭环基础上，新增“财富流动日志”面板。

v1.4.3 的核心方向：

- 保持本地优先。
- 继承 v1.4.2 的首次启动扫描、启动面板、六个子系统地图、个人生态系统闭环、能量管理系统闭环、财务管理系统入口、“我的钱包”、“收入来源”和“日常开销池”。
- 在财务管理系统中新增“财富流动日志”面板。
- 财富流动日志是本地、追加式、可扩展的资金事件记录层。
- 财富流动日志只记录 v1.4.3 之后发生的新事件。
- 财富流动日志只记录四类事件：入账、划款、消费和消费回退。
- 收入来源手动入账成功后生成入账日志。
- 从钱包容器划入日常开销池成功后生成划款日志。
- 日常开销池消费扣款成功后生成消费日志。
- 删除消费流水并把金额加回开销池后生成消费回退日志。
- 消费回退通过追加新日志表达，不删除或静默改写原消费日志。
- 消费回退日志应关联原消费日志或原消费流水。
- 财富流动日志不作为钱包余额或日常开销池余额的唯一事实来源。
- 不用财富流动日志重算钱包容器余额。
- 不用财富流动日志重算日常开销池余额。
- 日常开销池用户侧取消“消费流水”可见板块。
- `DailyExpenseEntry` 可以继续作为内部数据存在，用于支撑消费回退和日志关联。
- 用户侧统一在“财富流动日志”面板查看 v1.4.3 之后发生的消费和消费回退。
- 财务管理系统页面顺序为：我的钱包、收入来源、日常开销池、财富流动日志、财务系统未来结构、返回启动面板入口。
- 财富流动日志永远放在所有已开放财务业务模块之后，财务系统未来结构之前。
- 如果未来开放梦想账户或金鹅账户，它们应位于财富流动日志之上。
- 财务系统未来结构不再展示财富流动日志后续开放。
- 梦想账户和金鹅账户仍保持后续开放。
- 重置本地数据时清空财富流动日志。
- 不追溯 v1.4.1 的历史入账。
- 不追溯 v1.4.2 的历史划款或消费流水。
- 不记录钱包容器直接编辑余额的变化。
- 不做完整记账系统、完整账本事件源重构、旧数据迁移、消费分类、预算统计、周期判断、自动提醒、自动预测、通用退款流水、钱包之间任意转账、梦想账户或金鹅账户业务。
- 不使用财务诊断、投资建议、预算警报、超支惩罚、财务健康评分、AI 分析、账号系统、云同步、云端数据库、银行连接或支付连接。
- 不恢复用户侧 JSON 导出、Markdown 导出或完整个人说明书编辑入口。

## v1.4.2 状态

v1.4.2 是财务管理系统的第二个增量业务版本，目标是在 v1.4 的“我的钱包”本地余额快照闭环和 v1.4.1 的“收入来源”手动入账闭环基础上，新增“日常开销池”面板。

v1.4.2 的核心方向：

- 保持本地优先。
- 继承 v1.4.1 的首次启动扫描、启动面板、六个子系统地图、个人生态系统闭环、能量管理系统闭环、财务管理系统入口、“我的钱包”和“收入来源”。
- 在财务管理系统中新增“日常开销池”面板。
- 日常开销池只有一个全局池。
- 日常开销池面板采用上中下结构：主信息和划款入口、消费结算、消费流水。
- 用户可以在主信息区选择当前划款来源钱包容器。
- 用户可以切换当前划款来源钱包容器，切换只影响下一次划款。
- 用户可以从当前划款来源钱包容器划入任意正数金额。
- 划款后来源钱包容器余额减少，日常开销池余额增加。
- 划款后“我的钱包”的当前总余额和环形图更新。
- 划款只保存最近一次划款金额、日期和来源容器名称快照。
- 不保存完整划款历史。
- 用户消费时必须输入消费金额和备注。
- 消费金额必须为正数，且不能超过开销池余额。
- 消费后开销池余额减少，并生成本地消费流水。
- 消费不修改钱包容器余额。
- 用户可以删除消费流水。
- 删除消费流水后开销池余额加回，钱包容器余额不变。
- 当前划款来源容器被删除后，开销池余额和消费流水保留，消费结算仍可用，划款入口禁用并提示重新选择来源容器。
- 不做多个开销池、开销分类池、周期字段、周期判断、自动提醒、自动预测、预算警报、超支惩罚、财务健康评分、消费建议、完整财富流动日志、梦想账户或金鹅账户业务。
- 不使用财务诊断、投资建议、AI 分析、账号系统、云同步、云端数据库、银行连接或支付连接。
- 不恢复用户侧 JSON 导出、Markdown 导出或完整个人说明书编辑入口。

## v1.4.1 状态

v1.4.1 是财务管理系统的第一个增量业务版本，目标是在 v1.4 的“我的钱包”本地余额快照闭环基础上，新增“收入来源”面板。

v1.4.1 的核心方向：

- 保持本地优先。
- 继承 v1.4 的首次启动扫描、启动面板、六个子系统地图、个人生态系统闭环、能量管理系统闭环和财务管理系统入口。
- 保持“我的钱包”的本地余额快照闭环。
- 在财务管理系统中新增“收入来源”面板。
- 用户可以新建、编辑和删除收入来源。
- 收入来源金额支持固定金额和不固定。
- 收入来源频率支持日结、周结、月结、季结、年结和不固定。
- 收入来源必须绑定一个已有钱包容器。
- 固定金额来源可以手动入账，入账后对应钱包容器余额增加。
- 不固定金额来源手动入账时，用户必须输入本次到账金额。
- 入账后“我的钱包”的当前总余额和环形图更新。
- 绑定的钱包容器被删除后，收入来源保留，但不能入账，并提示重新选择流入钱包。
- 不记录每次入账历史。
- 不做流水账、支出记录、财富流动日志、日常开销池、梦想账户或金鹅账户业务。
- 不使用自动到账、到账提醒、收入预测、职业建议、财务诊断、投资建议、预算警报、超支惩罚、AI 分析、账号系统、云同步、云端数据库、银行连接或支付连接。
- 不恢复用户侧 JSON 导出、Markdown 导出或完整个人说明书编辑入口。

v1.4 的核心方向：

- 保持本地优先。
- 继承 v1.3 的首次启动扫描、启动面板、六个子系统地图、个人生态系统闭环和能量管理系统闭环。
- 将“财务管理系统”从后续开放入口推进为可进入状态。
- 财务管理系统真正可用的业务只有“我的钱包”。

- 用户直接编辑资金容器当前余额。

- 不做资金容器分类、记账流水、梦想账户、金鹅账户、金钱流入来源、日常开销池或财富流动日志业务。

## v1.0 状态

v1.0 是 LifeOS 的第一个本地可运行版本，目标是通过 9 道 onboarding 问题生成初始个人说明书控制面板。

从后续产品方向看，v1.0 应被理解为早期启动原型：它验证了本地优先、首次引导、确定性规则生成、编辑、导出和重置等基础能力。它不代表 LifeOS 的长期产品重心。

LifeOS 的长期主体应落在六个子系统上：

- 个人生态系统：生理基础与生活环境。
- 能量管理系统：心理余量与恢复。
- 认知管理系统：信息、学习、判断和反思。
- 人生目标管理系统：方向、项目和行动。
- 人际关系管理系统：连接、沟通和边界。
- 财务管理系统：资源、消费和自由度。

首次启动的 9 道 onboarding 问题只作为初始扫描和子系统推荐器。个人说明书应作为后期独立板块，由六个子系统的长期使用情况沉淀生成，而不是 LifeOS 的长期主体。

v1.0 的核心边界：

- 本地优先。
- 不使用账号系统。
- 不使用云同步。
- 不使用云端数据库保存个人生活数据。
- 不使用 AI 分析、AI 推理或心理诊断。
- 以确定性规则生成个人说明书初稿。
- 所有观察都标记为待验证观察。

## 工作规则

开发当前版本时，以 `doc/v1.4.3/design.md` 和 `doc/v1.4.3/tasks/` 为准。

如需理解历史实现背景，可以参考 `doc/v1.4.2/design.md`、`doc/v1.4.2/tasks/`、`doc/v1.4.1/design.md`、`doc/v1.4.1/tasks/`、`doc/v1.4/design.md`、`doc/v1.4/tasks/`、`doc/v1.3/design.md`、`doc/v1.3/tasks/`、`doc/v1.2/design.md`、`doc/v1.2/tasks/`、`doc/v1.1/design.md`、`doc/v1.1/tasks/`、`doc/v1.0/design.md` 和 `doc/v1.0/tasks/`，但不要把历史版本中已经过时的产品范围带入 v1.4.3。

如果发现设计文档、任务文件和代码实现不一致：

1. 先判断是否属于文档错误、实现偏差或需求变化。
2. 当前版本文档错误可以在 v1.4.3 文档中修正。
3. 实现偏差应优先修代码，使其回到 v1.4.3 边界内。
4. 需求变化不要直接写进 v1.4.3，应先进入 `doc/roadmap/` 或新的版本目录。

## 后续版本

每个正式版本应新建独立目录，例如：

```text
doc/v2.0/
```

未来版本可以引用 v1.0 的设计，但不应直接覆盖 v1.0 的历史文档。

在规划后续版本时，应优先纠正 v1.0 的产品重心偏移：不要继续围绕 onboarding 或早期说明书扩展细节，而应逐步建立六个子系统的主界面、数据模型和真实使用闭环。
