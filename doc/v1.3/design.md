# LifeOS v1.3 设计文档

## 1. 版本定位

LifeOS v1.3 是第二个真实子系统闭环版本。

v1.2 已经在六系统启动面板基础上开启了个人生态系统，并完成“记录 -> 本地保存 -> 回顾”的最小观察闭环。v1.3 不继续扩展 onboarding，不恢复个人说明书、导出或云端能力，也不修改 v1.2 的历史边界，而是开启第二个可反复使用的子系统：能量管理系统。

v1.3 的目标是：

> 建立能量管理系统的最小观察闭环。

这个闭环不是心理诊断、情绪分析、效率管理、任务规划或恢复建议系统。它帮助用户在本地记录和回顾心理余量、压力负载、情绪天气、注意带宽、社交电量和行动阻力。

## 2. 继承自 v1.2 的原则

v1.3 继续继承这些原则：

- 本地优先。
- 用户数据保存在用户自己的设备上。
- 不使用账号系统。
- 不使用云同步。
- 不使用云端数据库保存个人生活数据。
- 不使用 AI 分析、AI 推理、心理诊断、人格诊断、情绪诊断或医疗建议。
- 技术栈继续使用 Next.js + React + TypeScript。
- 本地持久化继续使用 IndexedDB + Dexie。
- 界面语言使用简体中文。
- 保留复古 Macintosh 风格和旧电脑感。
- onboarding 仍然只是初始扫描和子系统推荐器。
- 个人说明书仍然是后期板块，不是当前主功能。
- 推荐、扫描线索和子系统回顾都必须是确定性逻辑，不得伪装成诊断结论。

v1.3 不继承 v1.2 中“只开启个人生态系统”的范围限制。v1.3 会把能量管理系统推进为可使用的真实子系统；认知管理系统、人生目标管理系统、人际关系管理系统和财务管理系统仍保持后续开放。

## 3. v1.3 新增内容

### 3.1 能量管理系统最小闭环

v1.3 新增能量管理系统页面。

最小闭环是：

```text
进入能量管理系统
  -> 看到当前能量状态
  -> 选择性记录一个能量观察点
  -> 本地保存
  -> 查看今日能量轨迹
  -> 查看能量风向标
```

用户可以只查看，不记录。系统不能在进入页面时强制弹出记录表，也不能把记录做成连续打卡压力。

同一次进入能量管理系统页面时，同一个维度最多保留一个新观察点。用户在同一页面会话里再次选择同一维度时，系统应覆盖刚刚记录的那一条，而不是不断追加挤满今日轨迹。用户返回启动面板并重新进入后，才可以为同一维度产生新的观察点。

### 3.2 能量语义观察点

v1.3 继续使用“语义观察点”作为轻量状态记录模型。

能量语义观察点是一条本地事件记录，表示用户在某个时刻对某个能量维度做了一次观察。

用户看到的是语义状态，例如：

- 有余量。
- 偏重。
- 阴晴不定。
- 大脑宕机。
- 完全封闭。
- 停摆。

系统内部可以把语义状态映射为 `internalScore`，用于本地确定性统计。内部分数不作为用户侧评分展示。

### 3.3 六个能量维度

v1.3 第一版包含六个维度，顺序固定为：

1. 当前余量。
2. 压力负载。
3. 情绪天气。
4. 注意带宽。
5. 社交电量。
6. 行动阻力。

维度必须配置化。新增维度应优先通过配置完成，而不是复制一套业务逻辑。

### 3.4 三个能量面板

能量管理系统页面包含三个核心面板：

1. 当前能量状态。
2. 今日能量轨迹。
3. 能量风向标。

当前能量状态展示每个维度的最近观察。今日能量轨迹展示当天 0 点以后产生的观察点。能量风向标展示过去 1 天、7 天、15 天、30 天的本地观察摘要。

## 4. v1.3 修改内容

### 4.1 启动面板入口

v1.2 启动面板中个人生态系统已经可进入，能量管理系统仍是后续开放入口。v1.3 将能量管理系统入口从“后续开放”推进为可进入状态。

要求：

- 点击能量管理系统进入能量管理系统页面。
- 能量管理系统被推荐时，推荐卡片也能进入同一页面。
- 个人生态系统仍然可进入。
- 认知管理系统、人生目标管理系统、人际关系管理系统和财务管理系统仍显示后续开放反馈。
- 从能量管理系统页面可以返回 LifeOS 启动面板。

### 4.2 本地数据

v1.3 在本地数据模块中新增能量观察点存储。

本地数据至少保存：

- onboarding 回答。
- 启动扫描结果。
- 个人生态系统观察点。
- 能量管理系统观察点。

重置本地数据时，应同时清空个人生态系统观察点和能量管理系统观察点。

### 4.3 当前版本文档

v1.3 开发期间，`doc/current.md` 指向 v1.3。

v1.2 文档保持历史版本边界，不因 v1.3 新需求而改写。

## 6. 产品流程

v1.3 的首次使用路径继承 v1.2：

1. 用户在本地打开 LifeOS。
2. 如果没有本地扫描记录，展示启动界面和 onboarding。
3. 用户完成 9 道题。
4. LifeOS 保存扫描回答。
5. LifeOS 通过确定性规则生成初始扫描结果。
6. LifeOS 展示启动面板。
7. 启动面板展示六个子系统入口。

v1.3 新增的能量管理系统路径：

1. 用户点击能量管理系统入口。
2. LifeOS 展示能量管理系统页面。
3. 用户看到六个能量维度的当前状态。
4. 用户可以选择任一维度进行一次观察记录。
5. LifeOS 保存观察点到 IndexedDB。
6. 页面更新当前能量状态。
7. 今日能量轨迹显示当天观察点。
8. 能量风向标显示过去 1 天、7 天、15 天、30 天的观察摘要。
9. 用户可以返回 LifeOS 启动面板。

## 7. 界面设计

### 7.1 能量管理系统页面

页面主标题：

```text
能量管理系统
```

系统定义：

```text
心理余量与恢复
```

核心区块：

1. 当前能量状态。
2. 今日能量轨迹。
3. 能量风向标。
4. 返回启动面板入口。

### 7.2 当前能量状态

当前能量状态展示六张维度卡片：

1. 当前余量。
2. 压力负载。
3. 情绪天气。
4. 注意带宽。
5. 社交电量。
6. 行动阻力。

每张卡片包含：

- 维度名称。
- 一句说明。
- 最近一次语义状态。
- 记录时间。
- 更新入口。

如果某个维度今天没有记录，应显示：

```text
今日未观察
```

如果某个维度从未记录，应显示：

```text
尚未观察
```

如果最近一次记录不是今天，可以展示最近记录日期，但不能暗示它代表今天。

### 7.3 记录观察点

用户点击某个维度的更新入口后，可以选择该维度的一个语义状态。

要求：

- 一次只更新一个维度。
- 不要求填写其他维度。
- 七个语义状态应呈现为一条从“支撑高”到“拖累高”的档位轴，而不是散落按钮。
- 可选备注可以保留到后续实现；如果 v1.3 实现备注，备注必须保存在本地。
- 保存后生成一个观察点。
- 同一次进入页面内，同一维度重复保存应覆盖本次进入刚生成的观察点。
- 用户可以删除误触生成的观察点。
- UI 不显示 `internalScore`。

### 7.4 今日能量轨迹

今日能量轨迹展示当天 0 点以后产生的观察点。

要求：

- 按维度分组。
- 每个观察点显示语义状态和时刻。
- 每个观察点提供删除入口，用于处理误触。
- 没有记录的时刻保持空白。
- 不对空白时段做插值。
- 不使用折线连接不连续的观察点。

“0 点以后清零”只表示今日视图换日，不表示删除历史数据。

### 7.5 能量风向标

能量风向标支持范围切换：

- 1 天。
- 7 天。
- 15 天。
- 30 天。

每个范围按维度展示：

- 观察次数。
- 平均语义状态。
- 低位观察次数。
- 最近一次观察。
- 一句确定性摘要。

摘要必须使用“观察”“记录”“线索”“这段记录里”一类语言，不能使用“诊断”“建议”“你应该”“说明你已经”一类语言。

## 8. 能量维度定义

### 8.1 内部状态

每个维度使用 7 档内部状态：

```text
+3 很支撑
+2 支撑
+1 略微支撑
 0 中性或可用
-1 略微拖累
-2 拖累
-3 明显拖累
```

`没观察` 不是 7 档之一，也不应写入平均值。没有观察就是没有数据。

### 8.2 当前余量

当前余量回答：我现在还有多少心理和行动电量。

| 内部状态 | 用户语义 |
| --- | --- |
| +3 | 很充足 |
| +2 | 有余量 |
| +1 | 够用 |
| 0 | 勉强可用 |
| -1 | 偏低 |
| -2 | 快见底 |
| -3 | 已透支 |

### 8.3 压力负载

压力负载回答：我现在背着的压力重不重。对这个维度来说，负载越轻越支撑，负载越重越拖累。

| 内部状态 | 用户语义 |
| --- | --- |
| +3 | 很轻 |
| +2 | 可承载 |
| +1 | 稍有压力 |
| 0 | 有点满 |
| -1 | 偏重 |
| -2 | 很重 |
| -3 | 过载 |

### 8.4 情绪天气

情绪天气回答：我的内在气候如何。这个维度使用天气感语言，不使用诊断或病理标签。

| 内部状态 | 用户语义 |
| --- | --- |
| +3 | 明亮 |
| +2 | 平稳 |
| +1 | 柔和 |
| 0 | 阴晴不定 |
| -1 | 敏感 |
| -2 | 低沉 |
| -3 | 麻木/失控 |

### 8.5 注意带宽

注意带宽回答：我现在的脑内空间和处理能力够不够。这个维度只观察当下能量状态，不提前实现认知管理系统。

| 内部状态 | 用户语义 |
| --- | --- |
| +3 | 清晰宽裕 |
| +2 | 能聚焦 |
| +1 | 基本在线 |
| 0 | 有点散 |
| -1 | 容易分心 |
| -2 | 很难处理 |
| -3 | 大脑宕机 |

### 8.6 社交电量

社交电量回答：我现在能不能和人互动，需要多少边界。这个维度只观察当下互动余量，不提前实现人际关系管理系统。

| 内部状态 | 用户语义 |
| --- | --- |
| +3 | 想连接 |
| +2 | 可以交流 |
| +1 | 少量可用 |
| 0 | 看情况 |
| -1 | 需要边界 |
| -2 | 不想说话 |
| -3 | 完全封闭 |

### 8.7 行动阻力

行动阻力回答：我想行动时顺不顺，有没有启动困难或抗拒。对这个维度来说，阻力越低越支撑，阻力越高越拖累。这个维度只观察启动摩擦，不提前实现人生目标管理系统。

| 内部状态 | 用户语义 |
| --- | --- |
| +3 | 很顺手 |
| +2 | 可以启动 |
| +1 | 慢慢能动 |
| 0 | 有点卡 |
| -1 | 难启动 |
| -2 | 抗拒 |
| -3 | 停摆 |

## 9. 数据模型草案

### 9.1 能量维度 ID

```ts
type EnergyDimensionId =
  | "currentCapacity"
  | "pressureLoad"
  | "emotionalWeather"
  | "attentionBandwidth"
  | "socialBattery"
  | "actionResistance";
```

### 9.2 语义值

```ts
type EnergyInternalScore = -3 | -2 | -1 | 0 | 1 | 2 | 3;

type EnergySemanticValue = {
  id: string;
  label: string;
  internalScore: EnergyInternalScore;
};
```

### 9.3 维度定义

```ts
type EnergyDimensionDefinition = {
  id: EnergyDimensionId;
  label: string;
  description: string;
  values: EnergySemanticValue[];
};
```

### 9.4 观察点

```ts
type EnergyObservation = {
  id: string;
  dimensionId: EnergyDimensionId;
  valueId: string;
  valueLabel: string;
  internalScore: EnergyInternalScore;
  observedAt: string;
  note?: string;
};
```

观察点必须保存 `valueLabel` 和 `internalScore` 快照，避免未来调整维度配置后重新解释历史记录。

### 9.5 能量风向标摘要

```ts
type EnergyCompassRange = "1d" | "7d" | "15d" | "30d";

type EnergyDimensionSummary = {
  dimensionId: EnergyDimensionId;
  observationCount: number;
  averageInternalScore: number | null;
  lowObservationCount: number;
  latestObservation: EnergyObservation | null;
  summaryLabel: string | null;
};
```

## 10. 本地时间规则

今日能量轨迹按浏览器本地日期分组。

原则：

- 今日从用户设备本地时间 00:00 开始。
- 1 天、7 天、15 天、30 天统计同样使用本地日期边界。
- 0 点后今日轨迹自然为空，但历史数据不删除。
- 不引入服务器时间。

## 11. 聚合规则

### 11.1 当前能量状态

每个维度取最近一次观察点。

展示时区分：

- 今日已观察。
- 最近观察不是今天。
- 从未观察。

### 11.2 今日能量轨迹

筛选 `observedAt` 位于本地今天范围内的观察点。

按维度分组，再按时间升序展示。

### 11.3 能量风向标

对指定范围内的观察点按维度分组。

每个维度计算：

- `observationCount`
- `averageInternalScore`
- `lowObservationCount`
- `latestObservation`
- `summaryLabel`

`lowObservationCount` 统计 `internalScore <= -1` 的观察点。

`summaryLabel` 由平均值映射回该维度最接近的语义状态。它不是诊断，只是这段记录的平均位置。

职责：

- 继续负责首次启动、onboarding、启动面板和重置流程。

- 处理从启动面板进入能量管理系统。

- 完整个人说明书入口。
- JSON 导出入口。
- Markdown 导出入口。

### 12.3 Energy Management 模块

职责：

- 定义能量维度配置。

不应包含：

- onboarding 题目。

职责：

不应包含：

- React UI。

### 12.5 Local Data 模块

职责：

- 保存 onboarding 回答。

- 删除能量观察点。
- 清空所有本地 LifeOS 数据。

不应作为用户侧能力提供：

- JSON 导出。
- Markdown 导出。

## 13. 测试策略

v1.3 需要覆盖：

- 能量维度配置测试。
- 能量观察点本地存储测试。
- 当前能量状态计算测试。
- 今日能量轨迹计算测试。
- 能量风向标聚合测试。
- 能量管理系统页面组件测试。
- 启动面板入口测试。

1. 新用户仍然可以完成首次启动扫描。

3. 启动面板仍然展示六个子系统。
4. 点击个人生态系统仍可以进入个人生态系统页面。
5. 点击能量管理系统可以进入能量管理系统页面。
6. 其他四个子系统仍显示后续开放反馈。
7. 能量管理系统展示六个维度。
8. 用户可以只记录一个能量维度。
9. 记录后当前能量状态更新。
10. 今日能量轨迹展示当天观察点。
11. 能量风向标支持 1 天、7 天、15 天、30 天范围。
12. UI 不展示内部分数。
13. UI 不强制用户记录。
14. UI 没有连续打卡、完成率或惩罚文案。
15. UI 没有心理诊断、情绪诊断、健康建议、医疗建议、AI 分析或任务规划文案。
16. 重置后清空 onboarding、启动扫描、生态观察点和能量观察点。
17. 不引入账号、云同步、云数据库或 AI 分析。

建议最终验证命令：

```bash
npm run test
npm run typecheck
npm run lint
npm run build
```

## 14. 已确认决策

以下决策适用于 v1.3：

1. v1.3 继续使用 Next.js + React + TypeScript。
2. 本地持久化继续使用 IndexedDB + Dexie。
3. 本地优先原则不变。
4. 六个子系统是 LifeOS 的长期主体。
5. v1.3 保持个人生态系统已完成闭环。
6. v1.3 开启能量管理系统的真实业务闭环。
7. 认知管理系统、人生目标管理系统、人际关系管理系统和财务管理系统仍保持后续开放。
8. 能量管理系统使用语义观察点模型。
9. 用户侧展示语义状态，不展示内部分数。
10. 能量风向标使用本地确定性聚合。
11. v1.3 不使用 AI 分析、诊断、建议、账号、云同步或云端数据库。
