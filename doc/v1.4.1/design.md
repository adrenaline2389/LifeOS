# LifeOS v1.4.1 设计文档

## 1. 版本定位

LifeOS v1.4.1 是财务管理系统的第一个增量业务版本。

v1.4 已经开启财务管理系统，并完成“我的钱包”的本地余额快照闭环。v1.4.1 不修改 v1.4 的历史边界，不把 v1.4 已明确排除的能力回写到 `doc/v1.4/`。v1.4.1 在 v1.4 的基础上新增第二个真实财务面板：

> 收入来源。

v1.4.1 不是完整记账系统、流水系统、收入预测系统、预算系统、职业建议系统或财务顾问。它只提供一个本地、手动、无审判的“收入来源 -> 手动入账 -> 我的钱包更新”闭环。

v1.4.1 真正新增的可用业务只有：

- 收入来源。

## 2. 继承自 v1.4 的原则

v1.4.1 继续继承这些原则：

- 本地优先。
- 用户数据保存在用户自己的设备上。
- 不使用账号系统。
- 不使用云同步。
- 不使用云端数据库保存个人生活数据。
- 不使用 AI 分析、AI 推理、财务诊断、投资诊断、职业判断、心理诊断或医疗建议。
- 技术栈继续使用 Next.js + React + TypeScript。
- 本地持久化继续使用 IndexedDB + Dexie。
- 界面语言使用简体中文。
- 保留复古 Macintosh 风格和旧电脑感。
- onboarding 仍然只是初始扫描和子系统推荐器。
- 个人说明书仍然是后期板块，不是当前主功能。
- 推荐、扫描线索、子系统展示和财务计算都必须是确定性逻辑，不得伪装成诊断、建议或自动判断。

v1.4.1 继续保持这些已完成能力：

- 个人生态系统 v1.2 闭环。
- 能量管理系统 v1.3 闭环。
- 启动面板和六个子系统入口。
- 财务管理系统可进入。
- “我的钱包”本地余额快照闭环。
- 重置本地数据时清空生态观察点、能量观察点和钱包容器。

## 3. v1.4.1 新增内容

### 3.1 收入来源面板

v1.4.1 在财务管理系统页面新增“收入来源”面板。

用户可以在这里维护常见收入入口，例如：

- 工资。
- 项目款。
- 咨询费。
- 平台收入。
- 临时收入。
- 其他收入方式。

每条收入来源包含：

- 名称。
- 金额模式。
- 频率模式。
- 流入钱包容器。
- 可选备注。
- 创建时间。
- 更新时间。

收入来源面板依赖“我的钱包”。用户必须先创建至少一个资金容器，才能新增收入来源。

### 3.2 手动入账

每条收入来源提供一个手动入账按钮。

固定金额来源：

- 点击手动入账后，直接把固定金额加入绑定的钱包容器。
- 入账后更新目标钱包容器的 `balance` 和 `updatedAt`。
- “我的钱包”的当前总余额和环形图按 v1.4 规则刷新。

不固定金额来源：

- 点击手动入账后，用户必须输入本次到账金额。
- 确认后，系统把本次到账金额加入绑定的钱包容器。
- 入账后更新目标钱包容器的 `balance` 和 `updatedAt`。
- “我的钱包”的当前总余额和环形图按 v1.4 规则刷新。

v1.4.1 不记录每次入账历史。手动入账只更新钱包容器余额。

### 3.3 金额模式和频率模式

v1.4.1 使用两个维度封装收入来源：

1. 金额是否固定。
2. 频率是否固定。

这两个维度共同覆盖四种收入形态：

1. 金额固定，频率固定。例如每月固定工资。
2. 金额固定，频率不固定。例如偶尔按固定价格接一次咨询。
3. 金额不固定，频率固定。例如每月分成、绩效或平台收入。
4. 金额不固定，频率不固定。例如临时项目、偶发收入或其他不稳定入口。

v1.4.1 不保存 `stability` 字段。稳定性不作为独立标签，也不由系统判断。

### 4.3 本地数据

重置本地数据时，应同时清空个人生态系统观察点、能量管理系统观察点、钱包容器和收入来源。

## 6. 产品流程

v1.4.1 的首次使用路径继承 v1.4：

1. 用户在本地打开 LifeOS。
2. 如果没有本地扫描记录，展示启动界面和 onboarding。
3. 用户完成 9 道题。
4. LifeOS 保存扫描回答。
5. LifeOS 通过确定性规则生成初始扫描结果。
6. LifeOS 展示启动面板。
7. 启动面板展示六个子系统入口。
8. 用户点击财务管理系统入口。
9. LifeOS 展示财务管理系统页面。

v1.4.1 新增的收入来源路径：

1. 用户进入财务管理系统。
2. 用户看到“我的钱包”和“收入来源”。
3. 如果还没有钱包容器，收入来源面板提示先创建钱包容器，并禁用新增入口。
4. 用户在“我的钱包”创建至少一个资金容器。
5. 用户点击收入来源面板的新增按钮。
6. 用户填写收入来源名称、金额模式、频率模式、流入钱包容器和可选备注。
7. LifeOS 保存收入来源到 IndexedDB。
8. 收入来源列表新增一项。
9. 用户点击某条收入来源的手动入账按钮。
10. 固定金额来源直接入账。
11. 不固定金额来源要求填写本次到账金额后入账。
12. LifeOS 更新目标钱包容器余额。
13. “我的钱包”的当前总余额和环形图刷新。

## 7. 界面设计

### 7.1 财务管理系统页面

页面主标题继续使用：

```text
财务管理系统
```

系统定义继续使用：

```text
资源、消费和自由度
```

核心区块：

1. 我的钱包。
2. 收入来源。
3. 财务系统未来结构。
4. 返回启动面板入口。

### 7.2 收入来源面板

“收入来源”展示用户手动维护的常见进账入口。

它包含：

- 面板标题。
- 新增收入来源入口。
- 无钱包容器时的提示。
- 收入来源列表。
- 每条收入来源的手动入账按钮。
- 每条收入来源的编辑入口。

文案应明确这是本地手动收入入口，例如：

```text
这些是你手动维护的本地收入来源。
```

不得暗示系统连接了银行、支付平台、雇主系统、税务系统或外部账户。

### 7.3 收入来源编辑

用户可以新建、编辑和删除收入来源。

新建或编辑时包含：

- 名称。
- 金额模式：固定金额或不固定。
- 固定金额值。
- 频率模式：日结、周结、月结、季结、年结或不固定。
- 流入钱包容器。
- 可选备注。

要求：

- 名称必填。
- 固定金额必须为正数。
- 不固定金额不保存默认金额。
- 频率只用于展示。
- 流入钱包容器必填。
- 流入钱包容器必须从当前钱包容器中选择。
- 保存后更新 `updatedAt`。
- 删除收入来源不影响钱包容器余额。
- 删除入口放在编辑表单中。
- UI 不使用预算警报、超支惩罚、职业建议、投资建议或财务诊断文案。

### 7.4 手动入账交互

固定金额来源：

- 用户点击收入来源右侧的手动入账按钮。
- 系统将固定金额加入目标钱包容器。
- 页面刷新“我的钱包”的当前总余额和环形图。

不固定金额来源：

- 用户点击收入来源右侧的手动入账按钮。
- 系统展示本次到账金额输入。
- 用户填写正数金额并确认。
- 系统将本次金额加入目标钱包容器。
- 页面刷新“我的钱包”的当前总余额和环形图。

目标容器缺失：

- 收入来源仍显示在列表中。
- 手动入账按钮禁用。
- UI 提示用户重新选择流入钱包。

### 7.5 无钱包容器状态

当钱包容器数量为 0 时：

- 收入来源面板展示提示。
- 新增收入来源入口禁用。
- 用户不能保存没有目标钱包容器的收入来源。

## 8. 数据模型草案

### 8.1 收入来源

```ts
type MoneyAmountPattern =
  | {
      kind: "fixed";
      amount: number;
    }
  | {
      kind: "variable";
    };

type MoneyFrequencyPattern =
  | {
      kind: "fixed";
      interval: "daily" | "weekly" | "monthly" | "quarterly" | "yearly";
    }
  | {
      kind: "variable";
    };

type MoneyInflowSource = {
  id: string;
  name: string;
  amountPattern: MoneyAmountPattern;
  frequencyPattern: MoneyFrequencyPattern;
  targetWalletContainerId: string;
  note?: string;
  createdAt: string;
  updatedAt: string;
};
```

### 8.2 目标钱包状态

目标钱包状态不需要写入数据库，可以由当前钱包容器列表即时计算：

```ts
type MoneyInflowTargetStatus = "available" | "missing";
```

如果 `targetWalletContainerId` 在当前钱包容器列表中存在，则为 `available`。否则为 `missing`。

### 8.3 不新增入账事件

v1.4.1 不新增以下类型：

```ts
type MoneyInflowEvent = never;
type Transaction = never;
type LedgerEntry = never;
```

实现时不应真的定义这些类型。这里的作用是明确：v1.4.1 不保存入账历史，不创建流水事件源。

## 9. 入账规则

### 9.1 固定金额入账

```text
amountPattern.kind === "fixed"
  -> amount = amountPattern.amount
  -> target.balance = target.balance + amount
  -> target.updatedAt = now
  -> 保存 target
  -> 重新计算钱包汇总
```

### 9.2 不固定金额入账

```text
amountPattern.kind === "variable"
  -> 用户输入本次到账金额
  -> amount > 0
  -> target.balance = target.balance + amount
  -> target.updatedAt = now
  -> 保存 target
  -> 重新计算钱包汇总
```

### 9.3 频率规则

频率只用于展示。

频率不用于：

- 自动到账。
- 到账提醒。
- 收入预测。
- 月收入折算。
- 年收入折算。
- 稳定性判断。

27. 不保存每次入账历史。
28. 不引入账号、云同步、云数据库、银行连接、支付连接或 AI 分析。
29. UI 不使用预算警报、超支惩罚、财务健康评分、投资建议、职业建议或财务诊断文案。

建议最终验证命令：

```bash
npm run test
npm run typecheck
npm run lint
npm run build
```

## 12. 已确认决策

以下决策适用于 v1.4.1：

1. v1.4.1 继续使用 Next.js + React + TypeScript。
2. 本地持久化继续使用 IndexedDB + Dexie。
3. 本地优先原则不变。
4. 六个子系统是 LifeOS 的长期主体。
5. v1.4.1 保持个人生态系统已完成闭环。
6. v1.4.1 保持能量管理系统已完成闭环。
7. v1.4.1 保持财务管理系统可进入。
8. v1.4.1 保持“我的钱包”余额快照闭环。
9. v1.4.1 新增“收入来源”真实业务面板。
10. v1.4.1 的收入来源必须绑定钱包容器。
11. v1.4.1 使用手动入账，不使用自动到账。

13. v1.4.1 不做流水账、收入预测、职业建议、财务诊断、账号、云同步或云端数据库。
