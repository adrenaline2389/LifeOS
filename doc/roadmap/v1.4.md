# LifeOS v1.4 Roadmap

## 版本定位

LifeOS v1.4 计划开启第三个真实子系统：财务管理系统。

v1.2 已经完成个人生态系统的最小观察闭环。v1.3 已经完成能量管理系统的最小观察闭环。v1.4 不修改 `doc/v1.3/` 的版本边界，也不把财务管理系统需求写回 v1.3。v1.4 的新需求先进入 roadmap，后续确认后再进入独立版本目录。

v1.4 的目标是：

> 建立财务管理系统的基础设施版本。

财务管理系统在 v1.4 中不是完整理财工具、记账工具、预算工具、投资工具或财务顾问。它先提供一个本地、手动、无审判的“我的钱包”闭环，让用户看见自己的资金容器、当前总余额和资源分布。

v1.4 真正可用的业务只有：

- 我的钱包。

其他财务业务只作为未来结构方向出现，不在 v1.4 实现业务闭环。

相关架构决策：

- `doc/decisions/0001-local-first.md`
- `doc/decisions/0002-nextjs-as-local-web-app.md`
- `doc/decisions/0003-six-subsystems-as-core.md`
- `doc/decisions/0004-semantic-observation-events.md`

## 继承自 v1.3 的边界

v1.4 继续继承：

- 本地优先。
- 用户数据保存在用户自己的设备上。
- 不使用账号系统。
- 不使用云同步。
- 不使用云端数据库保存个人生活数据。
- 不使用 AI 分析、AI 推理、财务诊断、投资诊断、心理诊断或医疗建议。
- 技术栈继续使用 Next.js + React + TypeScript。
- 本地持久化继续使用 IndexedDB + Dexie。
- 界面语言使用简体中文。
- 保留复古 Macintosh 风格和旧电脑感。
- onboarding 仍然只是初始扫描和子系统推荐器。
- 个人说明书仍然是后期板块，不是当前主功能。
- 子系统展示必须是本地确定性逻辑，不得伪装成诊断、建议或自动判断。

v1.4 会把财务管理系统从“后续开放”推进为可进入状态。个人生态系统和能量管理系统继续保持可进入。认知管理系统、人生目标管理系统和人际关系管理系统仍保持后续开放。

## 产品心态

一个想开启财务管理系统的人，不一定是“不会理财”的人，而是想重新获得清晰感和选择权的人。

v1.4 的财务管理系统应避免把用户推向羞耻、焦虑或被训诫的状态。它不应该告诉用户“你应该怎么花钱”，也不应该用评分、警报或惩罚语言包装用户的财务状态。

v1.4 的第一性目标是：

> 帮助用户在本地、安全、无审判的环境里，看见自己的当前资源位置。

## 核心闭环

v1.4 的核心使用路径：

1. 用户从 LifeOS 启动面板进入财务管理系统。
2. 用户看到财务管理系统页面。
3. 用户看到“我的钱包”面板。
4. 用户新建一个资金容器。
5. 用户直接填写或编辑该容器的当前余额。
6. LifeOS 将钱包容器保存在本地 IndexedDB。
7. 页面更新当前总余额。
8. 当总余额为正时，页面展示彩色环形分布。
9. 当总余额为零或负时，页面展示灰色状态，不展示占比。
10. 用户可以继续编辑、删除或新增资金容器。

最小闭环是：

```text
进入财务管理系统
  -> 看到“我的钱包”
  -> 新建资金容器
  -> 填写当前余额
  -> 本地保存
  -> 当前总余额和环形图更新
  -> 用户可以直接编辑余额
```

v1.4 使用余额快照模型，不使用流水账模型。用户直接编辑容器当前余额，系统不追踪每一笔变化。

## 产品结构

### 面板 A: 我的钱包

“我的钱包”是 v1.4 唯一真正可用的业务。

它是一个手动余额快照工具。用户创建若干资金容器，填写余额；LifeOS 汇总当前总额，并在总额为正时展示彩色分布，在总额为零或负时展示灰色状态。

每个资金容器包含：

- 名称。
- 当前余额。
- 颜色。
- 可选备注。
- 创建时间。
- 更新时间。

v1.4 不给资金容器做分类。资金容器不区分银行卡、支付钱包、现金、信用负债、梦想账户或金鹅账户。用户只是在本地维护几个“钱的位置”。

### 面板 B: 财务系统未来结构

财务管理系统页面可以展示未来业务结构，但这些结构在 v1.4 只能作为后续开放提示。

后续开放结构包括：

- 金鹅账户。
- 梦想账户。
- 金钱流入来源。
- 日常开销池。
- 财富流动日志。

这些后续开放区块不能在 v1.4 中提供编辑、保存、计算、建议或业务闭环。

## 我的钱包规则

### 资金容器

资金容器是 v1.4 的核心本地数据对象。

建议概念模型：

```ts
type WalletContainer = {
  id: string;
  name: string;
  balance: number;
  color: string;
  note?: string;
  createdAt: string;
  updatedAt: string;
};
```

实现时可以按现有类型风格调整命名，但必须保留这些概念：

- 容器 ID。
- 容器名称。
- 当前余额。
- 展示颜色。
- 可选备注。
- 创建时间。
- 更新时间。

`balance` 允许为正数、零或负数。

### 总余额

当前总余额由所有资金容器的 `balance` 加总得到。

```text
totalBalance = sum(container.balance)
```

总余额只是用户手动维护的本地余额快照，不代表银行真实余额，不代表资产诊断，也不代表系统自动同步结果。

### 正余额状态

当 `totalBalance > 0`：

- 环形图正常展示。
- 只使用正余额容器计算占比。
- 中间显示“当前总余额”。
- 图例展示参与分布的容器名称、颜色、金额和占比。
- 余额为 0 或负数的容器不参与占比计算，但仍在容器列表中显示真实余额。

### 零余额状态

当 `totalBalance === 0`：

- 不展示占比。
- 环形图使用中性灰色空环或占位环。
- 中间显示“当前总余额 0”。
- 可以显示克制说明：“还没有可分配余额快照。”
- 不使用负面提醒或惩罚文案。

### 负余额状态

当 `totalBalance < 0`：

- 环形图整体变灰。
- 不展示各容器占比。
- 中间显示“当前负债金额”。
- 容器列表仍然显示每个容器的真实余额。
- 可以显示克制说明：“当前总和为负，暂不计算资金占比。”
- 不出现“警报”“危险”“你超支了”“你应该”等文案。
- 当用户更新容器后 `totalBalance > 0`，自动恢复正常彩色环形图和占比。

## 后续版本规划

v1.4 系列应按“小步开放一个真实财务面板”的方式推进。每个小版本都必须保持本地优先，并明确自己不是完整理财系统、预算系统、投资系统、财务诊断系统或 AI 分析系统。

不必为每个小版本单独维护一份 roadmap。v1.4 系列的候选方向统一收敛在本文件中；当某个小版本进入正式开发时，再新建对应版本目录，例如 `doc/v1.4.2/`，并以该目录中的 `design.md`、`prompt.md` 和 `tasks/` 作为执行依据。

### v1.4.1: 收入来源（已落地）

v1.4.1 已经在 v1.4 的“我的钱包”余额快照闭环上新增第二个真实财务面板：

> 收入来源。

正式版本边界以 `doc/v1.4.1/design.md` 和 `doc/v1.4.1/tasks/` 为准。独立的 `doc/roadmap/v1.4.1.md` 已不再提供额外决策价值，可以删除或归档；后续不应继续维护它作为当前需求来源。

v1.4.1 的核心闭环：

```text
进入财务管理系统
  -> 看到“收入来源”
  -> 新增收入来源
  -> 绑定一个钱包容器
  -> 用户手动确认一次入账
  -> 对应钱包容器余额增加
  -> 我的钱包总额和环形图更新
```

v1.4.1 的已确认边界：

- 收入来源必须绑定已有钱包容器。
- 金额支持固定金额和不固定金额。
- 频率支持日结、周结、月结、季结、年结和不固定。
- 频率只用于展示，不自动到账、不提醒、不预测。
- 固定金额来源可以直接手动入账。
- 不固定金额来源入账时必须输入本次到账金额。
- 入账只更新目标钱包容器余额，不记录每次入账历史。
- 绑定的钱包容器被删除后，收入来源保留，但不能入账，并提示重新选择流入钱包。

v1.4.1 不包含：

- 入账历史。
- 流水账。
- 支出记录。
- 财富流动日志。
- 日常开销池。
- 收入预测。
- 职业建议。
- 财务诊断。
- 投资建议。
- 预算警报。
- 超支惩罚。
- 银行连接、支付连接、账号系统、云同步、云端数据库或 AI 分析。

### v1.4.2: 日常开销池（已落地）

v1.4.2 已经在 v1.4.1 的“我的钱包 + 收入来源”闭环上新增第三个真实财务面板：

> 日常开销池。

正式版本边界以 `doc/v1.4.2/design.md` 和 `doc/v1.4.2/tasks/` 为准。

它不是完整预算系统，也不是完整财富流动日志。它已经落地一个本地、手动、无审判的“钱包容器 -> 划入开销池 -> 消费扣减 -> 本地消费流水”的闭环。

v1.4.2 采用方案 A：一个全局日常开销池，只记录当前池状态、最近一次划款信息和消费流水。

核心闭环：

```text
进入财务管理系统
  -> 看到“日常开销池”
  -> 在主信息区选择当前划款来源钱包容器
  -> 输入划入金额
  -> 钱包容器余额减少
  -> 日常开销池余额增加
  -> 用户输入消费金额和必填备注
  -> 开销池余额减少
  -> 生成本地消费流水
  -> 删除消费流水时回滚开销池余额
```

页面结构：

1. 我的钱包。
2. 收入来源。
3. 日常开销池。
4. 财务系统未来结构。
5. 返回启动面板入口。

“日常开销池”面板内部采用上中下结构：

1. 上方小窗口：开销池主信息和划款入口。
2. 中间小窗口：消费结算。
3. 下方小窗口：消费流水。

上方主信息和划款入口包含：

- 开销池当前余额。
- 当前划款来源钱包容器。
- 切换划款来源容器的入口。
- 划入金额输入。
- 划款按钮。
- 最近一次划款金额。
- 最近一次划款日期。
- 最近一次划款来源容器名称快照。

划款规则：

- 日常开销池只有一个。
- 用户可以在上方主信息区切换当前划款来源容器。
- 切换来源容器只影响下一次划款，不影响开销池当前余额、不影响消费流水、不回滚历史划款。
- 每次划款时，从当前选择的钱包容器扣钱，并把金额加到开销池。
- 划入金额必须是正数。
- 划入金额不能超过当前选择的钱包容器余额。
- 如果来源容器余额不足，不能划入。
- 不保存完整划款历史，只保存最近一次划款金额、日期和来源容器名称快照。
- 如果当前划款来源容器被删除，开销池余额和消费流水保留，消费结算仍可用；划款入口禁用，并提示重新选择划款来源容器。

中间消费结算包含：

- 消费金额输入。
- 消费备注输入。
- 立即扣款按钮。

消费规则：

- 消费只扣开销池余额，不直接读取或修改钱包容器。
- 消费金额必须是正数。
- 消费金额不能超过开销池余额。
- 如果开销池余额不足，不能消费。
- 备注必填。
- 备注首尾空格会被裁剪，裁剪后不能为空。
- 扣款成功后生成一条本地消费流水。
- 不允许负数消费。
- v1.4.2 不做退款、返还或反向流水。

下方消费流水包含：

- 消费金额。
- 备注。
- 消费时间。
- 创建时间。
- 删除入口。

消费流水规则：

- 消费流水保存在本地 IndexedDB。
- 删除消费流水时，把对应金额加回开销池余额。
- 删除消费流水不修改钱包容器余额，因为钱包容器只在划款时扣过一次。
- 消费流水是日常开销池面板内部的本地消费记录，不等同于完整财富流动日志。

数据模型草案：

```ts
type DailyExpensePool = {
  id: "default";
  balance: number;
  selectedWalletContainerId?: string;
  lastTransferAmount?: number;
  lastTransferAt?: string;
  lastTransferWalletContainerId?: string;
  lastTransferWalletContainerNameSnapshot?: string;
  createdAt: string;
  updatedAt: string;
};

type DailyExpenseEntry = {
  id: string;
  amount: number;
  note: string;
  spentAt: string;
  createdAt: string;
  updatedAt: string;
};
```

实现时可以按现有类型风格调整命名，但必须保留这些概念：

- 单一开销池。
- 当前余额。
- 当前划款来源钱包容器。
- 最近一次划款信息。
- 本地消费流水。
- 消费流水删除回滚开销池余额。

v1.4.2 不包含：

- 多个开销池。
- 开销分类池。
- 周期字段。
- 周期判断。
- 自动提醒。
- 自动预测。
- 预算警报。
- 超支惩罚。
- 财务健康评分。
- 消费建议。
- 财务诊断。
- 完整财富流动日志。
- 划款历史。
- 退款流水。
- 返还流水。
- 银行连接、支付连接、账号系统、云同步、云端数据库或 AI 分析。

### v1.4.3 已完成: 财富流动日志

状态：已实现并通过测试、类型检查、lint 和 build。

v1.4.3 在 v1.4.2 的“我的钱包 + 收入来源 + 日常开销池”闭环上新增第四个真实财务面板：

> 财富流动日志。

v1.4.3 的财富流动日志不是完整记账系统，也不是完整账本事件源。它是在现有业务动作之上新增一个本地、追加式、可扩展的资金事件记录层，用于记录从 v1.4.3 起发生的关键财富流动，并为后续梦想账户和金鹅账户的财富流转记录预留结构。

v1.4.3 只记录四类事件：

1. 入账：收入来源手动入账成功后生成日志。
2. 划款：从钱包容器划入日常开销池成功后生成日志。
3. 消费：日常开销池消费扣款成功后生成日志。
4. 消费回退：从财富流动日志中对原消费执行回退，把金额加回开销池并生成日志。

v1.4.3 的关键边界：

- 只记录 v1.4.3 之后发生的新事件。
- 不追溯 v1.4.1 的历史入账。
- 不追溯 v1.4.2 的历史划款或消费流水。
- 不记录用户直接编辑钱包容器余额的变化。
- 不把财富流动日志作为钱包余额或开销池余额的唯一事实来源。
- 不用日志重算钱包容器余额。
- 不用日志重算日常开销池余额。
- 不删除或静默改写既有日志。
- 消费回退通过追加一条“消费回退”日志表达，而不是修改原消费日志。
- 消费回退日志关联原消费备注，避免向用户展示内部事件 id。
- 原消费被回退后，回退按钮显示“已回退”并禁用，避免重复回退。

页面结构：

1. 我的钱包。
2. 收入来源。
3. 日常开销池。
4. 财富流动日志。
5. 财务系统未来结构。
6. 返回启动面板入口。

财富流动日志的长期位置规则：

- 财富流动日志永远放在所有已开放财务业务模块之后。
- 财富流动日志永远放在财务系统未来结构之前。
- 如果未来开放梦想账户，梦想账户应位于财富流动日志之上。
- 如果未来开放金鹅账户，金鹅账户应位于财富流动日志之上。
- 财富流动日志是已开放财务模块的统一记录层，不抢业务模块的位置。

日常开销池在 v1.4.3 中的调整：

- 用户侧取消日常开销池内部的“消费流水”可见板块。
- 日常开销池保留主信息和划款入口。
- 日常开销池保留消费结算入口。
- 日常开销池仍可显示最近一次划款信息。
- v1.4.2 已存在的 `DailyExpenseEntry` 可以作为内部数据继续保留，用于支撑消费回退。
- 用户侧统一在“财富流动日志”面板查看消费和消费回退记录。

已落地的财富流动日志交互：

- 日志按发生时间倒序展示。
- 每页最多展示 10 条日志，超过 10 条后通过上一页/下一页翻页。
- 入账和消费回退金额使用绿色徽标。
- 划款金额使用黄色徽标。
- 消费金额使用红色徽标。
- 单条日志采用紧凑行布局：左侧展示事件名称和日期，中间展示来源、目标、备注和关联原消费，金额固定在右上角。
- 消费回退不会修改或删除原消费日志，只追加一条消费回退日志。

建议事件模型草案：

```ts
type WealthFlowEventType =
  | "income_received"
  | "daily_expense_transfer"
  | "daily_expense_spent"
  | "daily_expense_refund";

type WealthFlowDirection = "in" | "out" | "transfer";

type WealthFlowSubjectType =
  | "wallet_container"
  | "income_source"
  | "daily_expense_pool"
  | "daily_expense_entry"
  | "dream_account"
  | "golden_goose_account";

type WealthFlowSubjectSnapshot = {
  type: WealthFlowSubjectType;
  id?: string;
  nameSnapshot: string;
};

type WealthFlowEvent = {
  id: string;
  type: WealthFlowEventType;
  direction: WealthFlowDirection;
  amount: number;
  occurredAt: string;
  source?: WealthFlowSubjectSnapshot;
  target?: WealthFlowSubjectSnapshot;
  relatedEventId?: string;
  relatedDailyExpenseEntryId?: string;
  note?: string;
  createdAt: string;
  updatedAt: string;
};
```

事件展示方向：

- 入账展示为 `+金额`。
- 划款展示为“划入开销池 金额”，并在详情中展示来源钱包和目标开销池。
- 消费展示为 `-金额`。
- 消费回退展示为 `+金额`，并关联原消费。

### v1.4.4 候选: 梦想账户

梦想账户用于承接“愿望”对象的财务侧信息。

未来人生目标管理系统可以在愿望相册里创建愿望。愿望可以作为跨系统数据块迁移到财务管理系统的梦想账户中。财务系统负责为愿望补充金额、优先级、已存入金额和完成状态。

建议未来拆成两个对象：

```ts
type Wish = {
  id: string;
  title: string;
  description?: string;
  originSystem: "life-goals" | "finance";
  createdAt: string;
  updatedAt: string;
  status: "seed" | "active" | "paused" | "fulfilled" | "archived";
};

type WishFinanceProfile = {
  wishId: string;
  targetAmount?: number;
  savedAmount?: number;
  priority: "high" | "medium" | "low";
  completedAt?: string;
  note?: string;
};
```

v1.4 不实现 `Wish` 或 `WishFinanceProfile`。这些只是后续设计方向。

### v1.4.5 候选: 金鹅账户

金鹅账户负责长期投资或长期不轻易动用的资源底座。

未来版本可以记录：

- 金鹅账户当前金额。
- 本次转入金额。
- 转入来源。
- 是否长期保留。
- 备注。

这个方向必须避免投资建议、收益率诊断、基金筛选和自动推荐。

## 验收标准

v1.4 完成时应满足：

- 新用户仍然可以完成首次启动扫描。
- 完成扫描后展示 LifeOS 启动面板。
- 启动面板展示六个子系统。
- 点击个人生态系统可以进入个人生态系统页面。
- 点击能量管理系统可以进入能量管理系统页面。
- 点击财务管理系统可以进入财务管理系统页面。
- 认知管理系统、人生目标管理系统和人际关系管理系统仍显示后续开放反馈。
- 财务管理系统展示“我的钱包”。
- 用户可以新建资金容器。
- 用户可以直接编辑资金容器当前余额。
- 用户可以删除资金容器。
- 资金容器余额允许为正数、零或负数。
- 当前总余额会根据资金容器自动汇总。
- 总余额为正时展示彩色环形分布。
- 总余额为零时展示灰色状态，不展示占比。
- 总余额为负时展示灰色状态，不展示占比，并显示当前负债金额。
- 负数容器不参与正余额占比计算，但仍在列表中显示。
- 用户更新余额后，环形图状态会按总余额正负自动切换。
- 钱包容器保存在本地 IndexedDB。
- 重置后清空 onboarding、启动扫描、生态观察点、能量观察点和钱包容器。
- 财务管理系统页面可以展示未来结构占位，但梦想账户、金鹅账户、流入来源、日常开销池和财富流动日志不能提供业务编辑或保存能力。
- UI 不使用预算警报、超支惩罚、财务健康评分、投资建议或财务诊断文案。
- UI 不重新引入 JSON 导出入口。
- UI 不重新引入 Markdown 导出入口。
- UI 不重新引入完整个人说明书编辑入口。
- 不引入账号、云同步、云数据库、银行连接、支付连接或 AI 分析。
