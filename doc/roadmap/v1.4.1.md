# LifeOS v1.4.1 Roadmap

## 版本定位

LifeOS v1.4.1 计划在 v1.4 的财务管理系统基础设施之上，新增第二个真实财务面板：

> 收入来源。

v1.4 已经完成“我的钱包”的本地余额快照闭环。v1.4.1 不修改 `doc/v1.4/` 的历史边界，也不把金钱流入来源写回 v1.4。v1.4.1 是后续版本规划：它让用户可以把常见收入来源记录为本地入口，并通过手动确认把一次收入加入指定的钱包容器。

这个版本不是完整记账系统、流水系统、收入预测系统、预算系统、职业建议系统或财务顾问。它只提供一个本地、手动、无审判的“收入来源 -> 手动入账 -> 我的钱包更新”闭环。

v1.4.1 真正可用的新增业务只有：

- 收入来源。

相关架构决策：

- `doc/decisions/0001-local-first.md`
- `doc/decisions/0002-nextjs-as-local-web-app.md`
- `doc/decisions/0003-six-subsystems-as-core.md`
- `doc/decisions/0004-semantic-observation-events.md`

## 继承自 v1.4 的边界

v1.4.1 继续继承：

- 本地优先。
- 用户数据保存在用户自己的设备上。
- 不使用账号系统。
- 不使用云同步。
- 不使用云端数据库保存个人生活数据。
- 不使用 AI 分析、AI 推理、财务诊断、投资诊断、职业判断、心理诊断或医疗建议。
- 技术栈继续使用 Next.js + React + TypeScript。
- 本地持久化继续使用 IndexedDB + Dexie。
- 界面语言使用简体中文。
- 保留复古 Macintosh 风格和旧电脑感。
- onboarding 仍然只是初始扫描和子系统推荐器。
- 个人说明书仍然是后期板块，不是当前主功能。
- 子系统展示和财务计算必须是本地确定性逻辑，不得伪装成诊断、建议或自动判断。

v1.4.1 必须保持 v1.4 已完成的“我的钱包”能力：

- 用户可以新建、编辑和删除资金容器。
- 用户可以直接编辑资金容器当前余额。
- 当前总余额根据资金容器自动汇总。
- 总余额为正时展示彩色环形分布。
- 总余额为零或负时展示灰色状态，不展示占比。
- 总余额为负时显示当前负债金额。
- 重置本地数据时清空钱包容器。

v1.4.1 不能破坏个人生态系统 v1.2 闭环、能量管理系统 v1.3 闭环、启动面板入口和本地重置流程。

## 产品心态

收入来源面板不是为了判断用户收入是否稳定，也不是为了告诉用户该换工作、该多赚钱或该如何理财。

它服务的是一种更温和的清晰感：

> 用户知道钱通常从哪里来，并能在钱到账时，用一个主动动作把它放进自己的本地钱包快照里。

系统不替用户预测收入，不替用户评价收入质量，不把收入包装成评分或诊断。频率和金额模式只是用户自己填写的事实结构，用来帮助识别收入来源。

## 核心闭环

v1.4.1 的新增核心路径：

1. 用户进入财务管理系统。
2. 用户看到已有的“我的钱包”面板。
3. 用户看到新增的“收入来源”面板。
4. 如果还没有钱包容器，收入来源面板提示先创建钱包容器，并禁用新增入口。
5. 用户点击收入来源面板的新增按钮。
6. 用户填写收入来源名称、金额模式、频率模式、流入钱包容器和可选备注。
7. LifeOS 将收入来源保存在本地 IndexedDB。
8. 收入来源列表新增一项。
9. 用户点击某条收入来源右侧的手动入账按钮。
10. 如果该来源是固定金额，LifeOS 直接把固定金额加入绑定的钱包容器。
11. 如果该来源是不固定金额，LifeOS 要求用户填写本次到账金额，再加入绑定的钱包容器。
12. “我的钱包”的目标容器余额、当前总余额和环形图按 v1.4 规则更新。

最小闭环是：

```text
进入财务管理系统
  -> 看到“收入来源”
  -> 新增收入来源
  -> 绑定一个钱包容器
  -> 用户手动确认一次入账
  -> 对应钱包容器余额增加
  -> 我的钱包总额和环形图更新
```

v1.4.1 不记录每一次入账历史。手动入账只更新目标钱包容器的余额。

## 产品结构

v1.4.1 的财务管理系统页面结构调整为：

1. 我的钱包。
2. 收入来源。
3. 财务系统未来结构。
4. 返回启动面板入口。

### 面板 A: 我的钱包

“我的钱包”继续保持 v1.4 已有业务边界。它是一个手动余额快照工具，不变成账户分类系统、资产负债表或流水账。

收入来源的手动入账会更新钱包容器余额，但不会改变钱包容器的数据模型含义：钱包容器仍然只是用户手动维护的“钱的位置”。

### 面板 B: 收入来源

“收入来源”是 v1.4.1 新增的真实业务面板。

用户可以在这里维护常见收入入口，例如：

- 工资。
- 项目款。
- 咨询费。
- 平台收入。
- 临时收入。
- 其他收入方式。

每条收入来源包含：

- 名称。
- 金额模式。
- 频率模式。
- 流入钱包容器。
- 可选备注。
- 创建时间。
- 更新时间。

收入来源列表中的每一项提供：

- 手动入账按钮。
- 编辑按钮。

删除入口放在编辑表单中。

### 面板 C: 财务系统未来结构

财务系统未来结构继续展示后续开放区块，但需要把“金钱流入来源”从后续开放结构中移出，或改为当前已开放的“收入来源”面板。

剩余后续开放结构包括：

- 日常开销池。
- 财富流动日志。
- 梦想账户。
- 金鹅账户。

这些后续开放区块不能在 v1.4.1 中提供编辑、保存、计算、建议或业务闭环。

## 收入来源规则

### 数据模型草案

```ts
type MoneyAmountPattern =
  | {
      kind: "fixed";
      amount: number;
    }
  | {
      kind: "variable";
    };

type MoneyFrequencyPattern =
  | {
      kind: "fixed";
      interval: "daily" | "weekly" | "monthly" | "quarterly" | "yearly";
    }
  | {
      kind: "variable";
    };

type MoneyInflowSource = {
  id: string;
  name: string;
  amountPattern: MoneyAmountPattern;
  frequencyPattern: MoneyFrequencyPattern;
  targetWalletContainerId: string;
  note?: string;
  createdAt: string;
  updatedAt: string;
};
```

实现时可以按现有类型风格调整命名，但必须保留这些概念：

- 来源 ID。
- 来源名称。
- 金额模式。
- 频率模式。
- 目标钱包容器 ID。
- 可选备注。
- 创建时间。
- 更新时间。

### 金额模式

金额从两个维度之一表达：

- 固定金额。
- 不固定金额。

固定金额必须填写正数金额。它表示每次用户点击手动入账按钮时，默认加入目标钱包容器的金额。

不固定金额不保存默认金额。用户每次点击手动入账按钮时，都必须填写“本次到账金额”，确认后才会加入目标钱包容器。

### 频率模式

频率从两个维度之一表达：

- 固定频率。
- 不固定频率。

固定频率支持：

- 日结。
- 周结。
- 月结。
- 季结。
- 年结。

频率只用于展示和识别收入来源，不触发自动到账、不生成提醒、不生成预测、不自动换算月收入或年收入。

### 四种收入来源形态

金额模式和频率模式共同封装四种常见收入形态：

1. 金额固定，频率固定。例如每月固定工资。
2. 金额固定，频率不固定。例如偶尔按固定价格接一次咨询。
3. 金额不固定，频率固定。例如每月分成、绩效或平台收入。
4. 金额不固定，频率不固定。例如临时项目、偶发收入或其他不稳定入口。

v1.4.1 不额外保存 `stability` 字段。稳定性不作为独立标签，也不由系统判断。

### 流入钱包容器

每条收入来源必须绑定一个目标钱包容器。

要求：

- 新增收入来源时，必须从已有钱包容器中选择一个。
- 如果没有任何钱包容器，不能新增收入来源。
- 如果绑定的钱包容器后来被删除，收入来源继续保留。
- 目标容器缺失时，该收入来源的手动入账按钮禁用。
- 目标容器缺失时，界面提示用户重新选择流入钱包。
- 目标容器状态不需要写入数据库，可以由当前钱包容器列表即时计算。

### 手动入账

手动入账是用户主动确认的一次操作。

固定金额来源：

```text
点击收入来源的入账按钮
  -> 读取固定金额
  -> 找到目标钱包容器
  -> target.balance += amount
  -> 更新 target.updatedAt
  -> 保存目标钱包容器
  -> 钱包总额和环形图刷新
```

不固定金额来源：

```text
点击收入来源的入账按钮
  -> 要求用户填写本次到账金额
  -> 校验金额为正数
  -> 找到目标钱包容器
  -> target.balance += amount
  -> 更新 target.updatedAt
  -> 保存目标钱包容器
  -> 钱包总额和环形图刷新
```

手动入账不创建流水记录，不创建事件历史，不提供撤销栈。用户如果误点固定金额来源，可以回到“我的钱包”手动修正目标容器余额。

## 本地数据

v1.4.1 需要在本地数据模块中新增收入来源存储。

本地数据至少保存：

- onboarding 回答。
- 启动扫描结果。
- 个人生态系统观察点。
- 能量管理系统观察点。
- 财务管理系统钱包容器。
- 财务管理系统收入来源。

建议新增本地数据能力：

```ts
getMoneyInflowSources(): Promise<MoneyInflowSource[]>;
saveMoneyInflowSource(source: MoneyInflowSource): Promise<void>;
deleteMoneyInflowSource(id: string): Promise<void>;
```

`clearLifeOSLocalData` 应扩展为同时清空收入来源。

v1.4.1 不提供用户侧 JSON 导出、Markdown 导出、Markdown 导入或备份恢复。

- 展示财务系统未来结构占位。

### Wallet Insights 模块

职责保持 v1.4 边界：

- 根据资金容器计算当前总余额。
- 根据资金容器计算钱包状态。
- 根据正余额容器计算分布占比。
- 所有逻辑必须本地确定性。

收入来源不应改变钱包汇总规则。手动入账只是更新某个钱包容器的 `balance`，随后继续使用 v1.4 已有汇总规则。

### Local Data 模块

职责扩展为：

- 保存收入来源。
- 读取收入来源。
- 删除收入来源。
- 清空所有本地 LifeOS 数据时同时清空收入来源。

不应作为用户侧能力提供：

- JSON 导出。
- Markdown 导出。
- 入账历史导出。

## 后续规划

v1.4.1 只开放收入来源。后续版本可以沿着以下方向逐步推进，但不应提前写入 v1.4.1 的实现范围。

### v1.4.2 候选: 日常开销池

让用户维护日常开销容器，例如房租、饮食、交通、学习、娱乐或其他生活支出池。

候选字段：

- 开销名称。
- 周期额度。
- 已用金额。
- 周期。
- 备注。

这个版本应避免预算惩罚、超支羞耻和强制完成率。

### v1.4.3 候选: 财富流动日志

记录收入、支出、储蓄、转入金鹅或转入梦想账户等资金流动事件。

候选字段：

- 类型。
- 金额。
- 流向。
- 发生时间。
- 关联钱包容器。
- 语义复盘。

财富流动日志可以成为后续财务统计的事件源，但不应在 v1.4.1 提前实现。

### v1.4.4 候选: 梦想账户

梦想账户用于承接“愿望”对象的财务侧信息。

未来人生目标管理系统可以在愿望相册里创建愿望。愿望可以作为跨系统数据块迁移到财务管理系统的梦想账户中。财务系统负责为愿望补充金额、优先级、已存入金额和完成状态。

建议未来拆成两个对象：

```ts
type Wish = {
  id: string;
  title: string;
  description?: string;
  originSystem: "life-goals" | "finance";
  createdAt: string;
  updatedAt: string;
  status: "seed" | "active" | "paused" | "fulfilled" | "archived";
};

type WishFinanceProfile = {
  wishId: string;
  targetAmount?: number;
  savedAmount?: number;
  priority: "high" | "medium" | "low";
  completedAt?: string;
  note?: string;
};
```

v1.4.1 不实现 `Wish` 或 `WishFinanceProfile`。这些只是后续设计方向。

### v1.4.5 候选: 金鹅账户

金鹅账户负责长期投资或长期不轻易动用的资源底座。

未来版本可以记录：

- 金鹅账户当前金额。
- 本次转入金额。
- 转入来源。
- 是否长期保留。
- 备注。

这个方向必须避免投资建议、收益率诊断、基金筛选和自动推荐。

## 验收标准

v1.4.1 完成时应满足：

- 新用户仍然可以完成首次启动扫描。
- 完成扫描后展示 LifeOS 启动面板。
- 启动面板展示六个子系统。
- 点击个人生态系统可以进入个人生态系统页面。
- 点击能量管理系统可以进入能量管理系统页面。
- 点击财务管理系统可以进入财务管理系统页面。
- 认知管理系统、人生目标管理系统和人际关系管理系统仍显示后续开放反馈。
- 财务管理系统仍展示“我的钱包”。
- 我的钱包仍支持新建、编辑和删除资金容器。
- 我的钱包仍根据资金容器自动汇总当前总余额。
- 我的钱包环形图仍按 v1.4 规则展示正余额、零余额和负余额状态。
- 财务管理系统新增“收入来源”面板。
- 没有钱包容器时，不能新增收入来源。
- 用户可以新增收入来源。
- 用户可以编辑收入来源。
- 用户可以删除收入来源。
- 收入来源名称必填。
- 收入来源金额支持固定金额和不固定。
- 固定金额必须是正数。
- 收入来源频率支持日结、周结、月结、季结、年结和不固定。
- 频率只用于展示，不自动到账、不提醒、不预测。
- 收入来源必须绑定一个钱包容器。
- 固定金额来源点击手动入账后，对应钱包容器余额增加。
- 不固定金额来源点击手动入账后，用户必须输入本次到账金额。
- 入账后“我的钱包”总余额更新。
- 入账后钱包环形图按现有规则更新。
- 绑定的钱包容器被删除后，收入来源仍保留。
- 目标容器缺失的收入来源不能入账，并提示重新选择流入钱包。
- 删除收入来源不影响钱包容器余额。
- 重置本地数据会清空收入来源。
- 不保存每次入账历史。
- 不引入账号、云同步、云数据库、银行连接、支付连接或 AI 分析。
- UI 不使用预算警报、超支惩罚、财务健康评分、投资建议、职业建议或财务诊断文案。
